# RSSI-в-beacon §15.1 — заход 1: потоковый режим content=4 (данные+метрики)

**Дата:** 2026-07-22. **Основание:** RECON_rssi_in_beacon.md (3dd6451) + рамка
архитектора (выбран путь `content=4`; команда 0x40 отвергнута — нет привязки к
источнику). **Цель захода 1:** прошивка отдаёт в потоке `content=4` тело принятого
кадра **вместе с метриками того же кадра** → хост сможет брать RSSI только из
партнёрских кадров (эхо-фильтр по `src_id`). **Станция+GUI — отдельный заход 2.**

Разбито на 3 малых шага (прошивка → хост-парсер → доки), пошаговые коммиты.

---

## Рамка архитектора (соблюдена)
- `content=4`: PAYLOAD = **метрики 30 Б** (`build_metrics_block`, как 1/2) + **`data_len`
  (u16 LE) + `data`** (тело `rx_frame` без FCS, `frame_len−2`, как 3). Порядок: сперва
  метрики, потом `data_len`+`data`. **CIR не включается.**
- `content=1/2/3` не трогаем. Guard как у `content=3`: `frame_len<2` или `>128` → кадр
  не слать (для `content=4` = нет валидных данных → нечего слать этим кадром).
- `SET_STREAM_MODE`: порог `mode>3 → mode>4`; присваивание `stream_content=mode` годится.
- `frame[320]` не расширять (макс `content=4` ~168 Б ≤ 320).
- CRC8/LEN16 — как в существующих ветках.

---

## Шаг 1 — прошивка `App\protocol\protocol.c` (коммит)
- `HandleSET_STREAM_MODE`: порог `mode > 3` → **`mode > 4`**; doxygen дополнен mode=4.
- `PROTOCOL_SendStreamFrame`: guard `frame_len<2/>RX_FRAME_MAX` расширен на
  `content==3 || content==4`; добавлена ветка `content==4`:
  `build_metrics_block(p)` (30 Б того же RXFCG-кадра) → `data_len (u16 LE)` →
  `memcpy(rx_frame, data_len)`. CIR не добавляется. `content=1/2/3` не изменены.
- **Проверка (моя):** `arm-none-eabi-gcc -fsyntax-only -Wall -Wextra` с флагами проекта —
  **чисто**. Приёмка на железе — за Ваганом.

## Шаг 2 — хост `tools\mks_stream.py` + `tools\mks_protocol.py` (коммит)
- `parse_stream_body`: ветка `content==4` — метрики `[5:35]` (`parse_signal_metrics`,
  как 1/2) + `data_len [35:37]` + `data [37:37+data_len]`; в возврат кладёт **и
  `metrics`, и `data`** (`cir=None`). Устойчивость к усечению (`ProtocolError`).
  `content=1/2/3` не изменены.
- `set_stream_mode`: допускает `mode=4`; заметка «командный канал занят потоком»
  **смягчена** до хостовой предосторожности (в прошивке лока команд НЕТ — различать
  ответ/поток по маркеру `SYNC 0xAA55` vs `SMARK 0xDECA`, по разведке §3/§6).
- **Проверка (моя):** py_compile всех tools + юниты: `content=4` (метрики final + data
  байт-в-байт), пустой data, усечение → `ProtocolError`, метрики `content=4` идентичны
  `content=2`, регресс `content=3/2`, `set_stream_mode(4)` — **все прошли**.

## Шаг 3 — доки (этот коммит)
- `PROTOCOL_SPEC.md` §6/§8: `mode=4` в строке команды; секция потока — `CONTENT=4`
  (формат метрики+`data_len`+`data`, без CIR, guard, привязка к одному RXFCG-кадру).
- Этот отчёт.

---

## Формат кадра content=4 (итог)
```
SMARK(0xDE 0xCA) | LEN16 | SEQ | DROPPED | CONTENT=4
                 | метрики30 | data_len(u16 LE) | data[data_len] | CRC8
```
Метрики (RSSI/FP_POWER/SNR + сырьё) — блок `build_metrics_block`, идентичный
`content=1/2`. `data` — тело принятого кадра без FCS, как `content=3`. Метрики и
данные — **один и тот же RXFCG-кадр** (в прошивке `rx_metrics.diag` и `rx_frame`
заполняются в одной ветке приёма). Максимум ~168 Б.

**Привязка RSSI к партнёру (зачем content=4, а не команда 0x40):** метрики едут ВМЕСТЕ
с телом каждого кадра. Станция (заход 2) применит эхо-фильтр по `src_id` из
прикладного заголовка внутри `data` — отбросит свои кадры (М2 слышит свой М1) и возьмёт
RSSI только из партнёрских. Команда 0x40 такой привязки не даёт (вернула бы метрики
«последнего» кадра, возможно своего эха; см. RECON §3).

## Приёмка захода 1 — на железе (ВАГАН), loopback M1→M2
Прошить свежую прошивку. Проверить приёмником (можно `mks_data_probe.py` не подойдёт —
он под `content=3`; для быстрой проверки — короткий скрипт/консоль, либо дождаться
станции захода 2). Минимальная проверка вручную:
1. `init → set_phy(Mode3) → rx_start → tx_periodic <payload> → set_stream_mode 4`.
2. Ожидаемо: приходят кадры `content=4`; в каждом:
   - **метрики физичны** — RSSI/FP_POWER/SNR как в `content=1` (loopback M1→M2, те же
     значения уровня);
   - **`data` байт-в-байт** равен переданному payload (без FCS).
3. Регресс: `content=1/2/3` не затронуты — `mks_stream_probe.py --content 1/2` и
   `mks_data_probe.py` (content=3) работают как прежде.

**Отчёт Вагана:** приходят ли `content=4`; метрики физичны (совпадают с `content=1`);
`data` совпадает; не сломаны ли `content=1/2/3`.
