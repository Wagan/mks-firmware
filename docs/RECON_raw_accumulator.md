# RECON — сырой мониторинг CIR-аккумулятора без RXFCG (§11 п.20)

**Тип:** разведка по документам/коду (без железа, правок нет). **Дата:** 2026-07-24.
**Вопрос:** можно ли снять CIR-аккумулятор, когда нормального приёма кадра (RXFCG) не
было — для будущего макета-обнаружителя. **Источники:** `Drivers\decadriver\deca_regs.h`
и `deca_device_api.h` (vendor-комментарии = парафраз UM), `App\protocol\protocol.c`,
`docs\reference\decawave-examples\`.

> ⚠️ **DW1000 User Manual (PDF) в репозитории ОТСУТСТВУЕТ** (в `docs\archive\` только
> «DW1000 Driver Library v1.3.01», MKS-доки — самого UM нет). Прямые §-цитаты UM дать
> НЕ могу; ниже — vendor-комментарии из `deca_regs.h`/`api.h` (они парафразируют UM) и
> код. Пункты, требующие UM, помечены **«СВЕРИТЬ С UM»**.

## 1. Что UM говорит о содержимом CIR-аккумулятора
- `ACC_MEM_ID = 0x25` — «Read access to accumulator data» (`deca_regs.h:660`).
- **СВЕРИТЬ С UM (в репо не найдено):** когда именно аккумулятор наполняется (в процессе
  накопления преамбулы / при SFD / только после декодирования), и есть ли явное
  утверждение, что данные валидны только при RXFCG. В vendor-комментариях такого
  утверждения нет; в API-доке `dwt_readaccdata` (`api.h:1255–1271`) **привязки к RXFCG
  НЕ указано** — сказано лишь: первый октет — dummy (внутренняя задержка чтения),
  отбрасывается; читает по смещению. (б)+(в): «валиден только при RXFCG» — не
  подтверждено и не опровергнуто источниками в репо.

## 2. События приёма без RXFCG (SYS_STATUS, `deca_regs.h:263–304`)
Биты (комментарии vendor):
- `RXPRD` 0x100 — **Preamble Detected** (`:263`) — корреляция преамбулы началась.
- `RXSFDD` 0x200 — **SFD Detected** (`:264`).
- `LDEDONE` 0x400 — **LDE processing done** (`:265`) — **LDE отработал → first path
  посчитан** (алгоритм leading-edge завершён).
- `RXPHD` 0x800 — PHY Header Detect (`:266`); `RXPHE` 0x1000 — PHY Header Error (`:267`).
- `RXDFR` 0x2000 — Data Frame Ready (`:268`); `RXFCG` 0x4000 — FCS **Good** (`:269`);
  `RXFCE` 0x8000 — FCS **Error** (`:270`, кадр принят, но CRC не сошёлся).
- `RXRFSL` 0x10000 — Reed-Solomon frame sync loss (`:272`); `RXSFDTO` 0x4000000 — SFD
  timeout (`:283`).
- Группы: `ALL_RX_GOOD = RXDFR|RXFCG|RXPRD|…` (`:297`); `ALL_RX_ERR = RXPHE|RXFCE|
  RXRFSL|RXSFDTO|…` (`:304`).

**Факт (по названиям бит):** `LDEDONE` и `RXFCE` означают, что приёмный тракт УЖЕ отработал
корреляцию/LDE (аккумулятор наполнен ходом приёма), при этом кадр либо не декодирован
до конца (LDEDONE без RXFCG), либо принят с битым CRC (RXFCE).
**СВЕРИТЬ С UM:** гарантирует ли UM валидность/полноту аккумулятора именно на этих
событиях (напр. «после LDEDONE first-path индекс достоверен», «при RXFCE accumulator
содержит CIR кадра») — в vendor-комментариях прямого утверждения нет.

## 3. Затирание аккумулятора
- Наш код (`protocol.c` район CaptureCIR): комментарий, что accumulator затирается при
  перевзводе приёма (`dwt_rxenable`), поэтому снимок берём ДО rxenable.
- **СВЕРИТЬ С UM (в репо не найдено):** формальное правило перезаписи — при старте нового
  приёма / при обнаружении новой преамбулы / иное. Vendor-комментарии `dwt_readaccdata`
  правил перезаписи НЕ описывают. Это определяет «успеем ли прочитать в сценарии без
  RXFCG» и требует UM/эксперимента.

## 4. Штатные режимы под задачу (API/UM)
- **SNIFF mode** (`dwt_setsniffmode`, `api.h:749–767`, ссылается на **UM §4.5
  «Low-Power SNIFF mode»**): приёмник секвенсируется ON/OFF для экономии — это про
  энергосбережение, **НЕ про доступ к аккумулятору без кадра**.
- **Continuous frame mode** (`dwt_configcontinuousframemode`, `api.h:1436–1448`) — это
  **continuous TX для тестов регуляторики**, не RX-аккумулятор.
- `dwt_setdblrxbuffmode` (`:829`) — двойной RX-буфер (не про аккумулятор без кадра).
- **Чтение аккумулятора/диагностики НЕ привязано в API к RXFCG:** `dwt_readaccdata`
  (`:1271`) и `dwt_readdiagnostics` (`:1311`) — просто читают память/регистры
  диагностики; в их doc-комментариях условия «только после RXFCG» НЕТ. Означает ли это,
  что вызов в любой ветке даст осмысленный CIR — **источники не утверждают** (в).
- **СВЕРИТЬ С UM:** есть ли штатный «диагностический/тестовый» режим приёмника с доступом
  к аккумулятору без успешного кадра (в vendor-API отдельной такой функции не видно).

## 5. Что у нас в коде сейчас
- **`PROTOCOL_CaptureCIR`** (protocol.c): читает окно аккумулятора `dwt_readaccdata`
  (dummy-байт + count×4), центр окна — `fp = rx_metrics.diag.firstPath >> 6`. Т.е.
  **привязан к `firstPath`**, а `firstPath` берётся из `dwt_readdiagnostics`, которая
  вызывается **ТОЛЬКО в ветке RXFCG** (`PROTOCOL_PollRadio`). Длина аккумулятора —
  `1016` (PRF64) / `992` (PRF16).
- **Вызов CaptureCIR — только в ветке RXFCG** (после `dwt_readdiagnostics`), т.е.
  сегодня снимок опирается на поле, заполняемое лишь при успешном кадре.
- **Ветка `ALL_RX_ERR`** (`PROTOCOL_PollRadio`): сейчас **снимает флаги ошибок,
  `dwt_rxreset()`, `dwt_rxenable()`** — **аккумулятор/диагностику НЕ читает**.
- **Кирпичи, которые уже есть:** `dwt_readaccdata` (порт SPI), `dwt_readdiagnostics`,
  опрос всех бит `SYS_STATUS` в main loop, ветка `ALL_RX_ERR` как точка перехвата
  «приём был, кадра нет». Чего нет: чтения аккумулятора/диагностики в НЕ-RXFCG-ветках и
  захвата CIR без валидного `firstPath` (напр. по фиксированному окну/энергии).

## 6. Референс EVK/DecaRanging
- `docs\reference\decawave-examples\ss_init_main.c` / `ss_resp_main.c`: опрос
  `SYS_STATUS` на `RXFCG | ALL_RX_TO | ALL_RX_ERR`; в ветке `ALL_RX_ERR` — **только
  снятие флагов** (`ss_init:171`, `ss_resp:191`), затем повтор. **Ни accumulator, ни
  diagnostics в этих примерах НЕ читаются вообще** (даже при RXFCG) — примеры про TWR,
  не про CIR. ⇒ **референса чтения аккумулятора вне RXFCG в наших примерах НЕТ.**

## Итог по разделам
**(а) Документировано (vendor-комментарии = парафраз UM; сам UM PDF в репо отсутствует):**
- Есть биты `SYS_STATUS`, отражающие стадии приёма без RXFCG: `RXPRD`, `RXSFDD`,
  `LDEDONE`, `RXPHD`, `RXPHE`, `RXFCE`, `RXRFSL`, `RXSFDTO`. `LDEDONE`/`RXFCE` по смыслу
  = корреляция/LDE отработали (аккумулятор наполнен ходом приёма).
- `dwt_readaccdata`/`dwt_readdiagnostics` в API **не заявляют** привязки к RXFCG.
- SNIFF (UM §4.5) — энергосбережение, не про аккумулятор без кадра.

**(б) Видно в коде:**
- CaptureCIR привязан к `firstPath` (только RXFCG); вызывается только в RXFCG.
- Ветка `ALL_RX_ERR` есть, но аккумулятор/диагностику не читает.
- Кирпичи чтения (`readaccdata`/`readdiagnostics`) готовы; захвата без `firstPath` нет.
- Референс-примеры аккумулятор не читают.

**(в) Неизвестно / требует UM или эксперимента на железе:**
- Валиден ли аккумулятор при `RXFCE`/после `LDEDONE` без `RXFCG` (полнота/достоверность CIR).
- Точное правило перезаписи аккумулятора (при rxenable / новой преамбуле / ином).
- Достоверность `firstPath`/LDE вне успешного кадра; можно ли снять осмысленное окно по
  фиксированному центру/энергии без `firstPath`.
- Есть ли в UM штатный диагностический режим приёмника с доступом к аккумулятору без кадра.

Вывод о реализуемости — за архитектором (по UM + при необходимости эксперименту).
