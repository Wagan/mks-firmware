## 2. Что чинить (механизм)

В прошивке есть программный снимок CIR `cir_snap` (после CIR-1) с полем `valid`, и
кэш метрик `rx_metrics` с полем `valid`. Они ставятся в 1 при приёме кадра и держат
последние данные. GET_CIR/GET_SIGNAL_METRICS отдают их, пока `valid==1` — даже если
новых кадров давно не было. Отсюда «залипший» старый CIR/метрики.

Правка: при СТАРТЕ приёма (RX_START) и при переинициализации (INIT) — обнулять эти
флаги валидности. Тогда до прихода ПЕРВОГО нового кадра GET_CIR/GET_SIGNAL_METRICS
вернут TIMEOUT (пусто), а не старый снимок. Экран консоли/GUI будет чистым, пока в
эфире нет сигнала; при первом реальном кадре снимок обновится и всё отобразится.

**НЕ трогать аппаратный аккумулятор** (никаких попыток «очистить ACC_MEM» —
не нужно и не делается вендором). Сбрасываем ТОЛЬКО программные флаги.

## 3. Правка (protocol.c)

Свериться с фактическими именами полей в коде репо. Ожидаемо:

### 3.1. В `HandleRX_START` (обработчик 0x30)

В начало обработки (после проверок готовности модуля, до/после `dwt_rxenable` —
логичнее ДО включения приёма) добавить сброс:
```c
    cir_snap.valid   = 0;   /* сбросить залипший снимок CIR — до первого нового кадра */
    rx_metrics.valid = 0;   /* и кэш метрик */
```
> Если в коде снимок/метрики названы иначе — использовать фактические имена.
> Если есть счётчики (напр. rx_metrics.count) — их НЕ обнулять без необходимости
> (count — накопительный счётчик кадров; сбрасывать его не просили). Обнуляем ТОЛЬКО
> флаги валидности, чтобы GET_CIR/GET_SIGNAL_METRICS вернули «нет данных» до первого
> нового кадра. Если сброс count желателен для чистого старта — вынести отдельно и
> НЕ делать в этом TASK без подтверждения (по no-guessing).

### 3.2. В `HandleINIT` (обработчик 0x01)

После цикла инициализации модулей (или в начале) — тоже обнулить флаги, чтобы
переинициализация давала чистое состояние:
```c
    cir_snap.valid   = 0;
    rx_metrics.valid = 0;
```

### 3.3. Проверить `PROTOCOL_Init`

Если в `PROTOCOL_Init` эти флаги уже обнуляются при старте прошивки — хорошо, ничего
не менять там. Правка нужна именно в RX_START/INIT (рантайм-сброс по команде).

## 5. Проверка на железе (ВАГАН, после ШАГА 0)

Сценарий «залипания» (в консоли, без источника сигнала):
1. Перепрошить свежий билд (ШАГ 0).
2. `python mks_console.py COM3`
3. `init` → OK. `mode3` (или mode4) → OK. `rxstart` → OK.
4. Подать сигнал (киты/передатчик в том же режиме) на пару секунд, затем `cir` →
   должен показать нормальный CIR с бугорком у FP. Убедиться, что приём был.
5. УБРАТЬ сигнал (выключить источник). Подождать. Снова `cir`:
   - Ожидаемо ДО правки: показывал старый бугорок (залипание).
   - Ожидаемо ПОСЛЕ правки: **если заново дать `rxstart`** → `cir` вернёт TIMEOUT
     (снимка нет), пока не придёт новый кадр. То есть после рестарта приёма без
     сигнала экран чистый.
6. Ключевая проверка: `rxstart` (повторный) при выключенном источнике → сразу `cir`
   → ожидаем **TIMEOUT/пусто** (не старый бугорок). Затем включить источник → `cir`
   → снова нормальный CIR. Это подтверждает сброс.
7. `metrics` после повторного `rxstart` без сигнала → тоже «нет приёма»/TIMEOUT
   (rx_metrics.valid сброшен), а не старые значения.

Отчёт: подтвердить, что после `rxstart` без сигнала cir/metrics не показывают старые
данные, а после появления сигнала — снова корректны.

> Примечание: если хочется, чтобы залипание чистилось БЕЗ повторного rxstart (т.е.
> «нет сигнала X секунд → само погасло») — это уже таймаут-логика, её место в GUI
> (поток молчит) или отдельная прошивочная фича с таймером. В этом TASK — только
> сброс по команде RX_START/INIT. Если Вагану нужно авто-гашение по таймауту —
> обсудим отдельно.
