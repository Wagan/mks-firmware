## 2. Последовательность старт/стоп (движок потока)

**Старт детекции** (кнопка, как сейчас): в фоновом воркере, чтобы не морозить UI:
```
INIT → SET_PHY_CONFIG(выбранный режим из PHY_MODES) → RX_START → SET_STREAM_MODE 1
→ перейти в режим ЧТЕНИЯ потока (фоновый поток читает потоковые кадры).
```
**Стоп детекции**: `SET_STREAM_MODE 0` → flush_input → `RX_STOP`. Остановить чтение.
Конфиг-кнопки (INIT/режим) блокировать во время активного потока (как сейчас).

content в Шаге 1 фиксирован = 1 (метрики+CIR). Полуширина окна CIR — дефолт прошивки.

## 3. Приём и разбор потока (переиспользовать код probe)

- Парсер потокового кадра уже есть в `mks_stream_probe.py` (поиск SMARK 0xDE 0xCA,
  чтение LEN16, SEQ, DROPPED, CONTENT, PAYLOAD, CRC-проверка, ре-синхронизация при
  сбое; разбор метрик 30 б и, при CONTENT=1, окна CIR). **Вынести этот парсер в общий
  модуль** (напр. дополнить `mks_protocol.py` функцией разбора потокового кадра ИЛИ
  создать `tools/mks_stream.py`), чтобы и probe, и GUI использовали ОДИН код.
  probe после рефактора должен работать как раньше (проверить).
- Фоновый поток GUI: в цикле читает из порта, склеивает кадры парсером, кладёт
  результат (метрики + CIR + SEQ/DROPPED) в `queue.Queue`. Главный поток по
  `root.after(~50 мс)` забирает ПОСЛЕДНИЙ доступный кадр для отрисовки, а для ЗАПИСИ
  обрабатывает ВСЕ пришедшие кадры (не только последний).
- Счётчики для телеметрии: fps (по времени), последний SNR/RSSI/FP_POWER, fp_index,
  DROPPED(fw), host_lost (дырки SEQ) — можно показывать (как в probe), но компактно.

## 4. Страховки от залипшего потока (ОБЯЗАТЕЛЬНО)

Три уровня (прошивочный уже сделан — INIT сбрасывает stream_active):
1. **Нормальный выход:** при закрытии окна (WM_DELETE_WINDOW) и при «Стоп» —
   `SET_STREAM_MODE 0` → flush → `RX_STOP` → закрыть порт. (Как сейчас, но добавить
   явный SET_STREAM_MODE 0.)
2. **При подключении:** сразу после открытия порта — flush_input, послать
   `SET_STREAM_MODE 0` (на случай, если плата осталась в потоке от прошлого раза),
   ещё раз flush, и только потом работать. Это защищает от «поток уже сыплется».
3. (Прошивка, уже готово: INIT гасит stream_active — GUI на это опирается, отдельно
   делать не нужно.)

Ошибки порта в потоке (SerialException/ProtocolError) — не ронять GUI: поймать,
показать в статусе, аккуратно остановить поток/детекцию.

## 7. Проверка на железе (ВАГАН)

Передатчик (плата 2, Mode 4, txperiodic) в эфире; приёмник (плата 1, COM3).
1. Запустить GUI, Подключить (COM3), INIT, режим Mode 4, Старт.
2. Ожидаемо: кружок зелёный, график CIR живой (обновляется плавно, ~10–20/с),
   телеметрия (fps ~200, SNR/RSSI физичны), DROPPED=0.
3. Выключить передатчик → через ~1 с кружок серый, график гаснет/чистится
   (гашение по таймауту). Включить → снова зелёный и живой график.
4. Запись: Полный → короткая запись → проверить metrics.csv + cir.csv (связь по
   frame_id, все кадры записаны — не прорежены).
5. Закрыть GUI, открыть консоль (COM3), init → OK, команды работают (поток погашен
   при выходе И страховкой INIT).
6. Аварийный тест: убить GUI (не через Стоп), открыть консоль, init → OK (страховка).
