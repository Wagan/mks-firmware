# RECON — однокабельный полудуплекс на одном модуле (§15.1 дальность)

**Тип:** разведка по коду (правки НЕ вносились). **Дата:** 2026-07-22.
**Гипотеза:** в станционном режиме собственный M1 (TX) глушит собственный M2 (RX) —
оба на одной плате, работают одновременно → −94 dBm на 2 м и обрывы, тогда как приём от
далёкого EVK (5 м) даёт −95 с рабочим CIR (RX-тракт исправен). **Идея:** новый режим —
полудуплекс через ОДИН модуль, второй выключен, TX/RX разнесены во времени (как EVK).
**Источник:** `App\protocol\protocol.c`, `App\board\board_config.h`,
`App\platform\deca_port.c`, `deca_device_api.h`, референс `ss_resp_main.c`. Только факты.

## 1. Насколько жёстко зашита раздельность M1/M2 (compile-time #define → runtime индекс)
`DW_TX_SOURCE_DEV = DW_DEV_M1` и `DW_RX_LISTEN_DEV = DW_DEV_M2` — **compile-time `#define`**
(`board_config.h:42,46`). Но `deca_port_select_device(idx)` — **рантайм**
(`deca_port.c:53`): `dwt_setlocaldataptr(idx)` + переключение активной SPI-шины/CS из
таблицы по индексу (`:44,:46`). Т.е. индекс устройства выбирается в рантайме; роли
зафиксированы только тем, что во ВСЕХ точках подставлен фиксированный `#define`.

**TX-путь жёстко берёт `DW_TX_SOURCE_DEV` (=M1):**
| Точка | protocol.c |
|---|---|
| `HandleTX_FRAME` — проверка/выбор | `:1050` / `:1051` |
| `HandleTX_STOP` — проверка/выбор | `:1083` / `:1084` |
| `HandleTX_PERIODIC` — проверка | `:1107` |
| `HandleSET_TX_POWER` — проверка/PGdelay/выбор | `:1154` / `:1157` / `:1160` |
| `PROTOCOL_PollTx` — выбор | `:1373` |

**RX-путь жёстко берёт `DW_RX_LISTEN_DEV` (=M2):**
| Точка | protocol.c |
|---|---|
| `HandleRX_START` — проверка/выбор | `:757` / `:758` |
| `HandleRX_STOP` — проверка/выбор | `:778` / `:779` |
| `PROTOCOL_PollRadio` — выбор | `:1309` |
| `build_metrics_block` — читает state (prf/channel) | `:884` / `:890` |
| `PROTOCOL_CaptureCIR` — читает prf | `:1269` |

**Вывод:** раздельность — **логическая** (через `#define`-индексы), НЕ аппаратный барьер.
Два чипа физически на РАЗНЫХ SPI/CS (M1→SPI2, M2→SPI3), но один чип может делать оба
направления, если TX-путь тоже выберет его индекс.

## 2. Может ли ОДИН модуль и передавать, и принимать
- **Аппаратно — да** (EVK: один чип делает оба, §EVK). В коде мешает только то, что
  TX-путь подставляет `DW_TX_SOURCE_DEV` (M1), а RX-путь — `DW_RX_LISTEN_DEV` (M2).
- Если `DW_TX_SOURCE_DEV == DW_RX_LISTEN_DEV` (оба = M2), `deca_port_select_device`
  выберет один и тот же чип для TX и RX → **однокабельный режим**. **Доказательство,
  что это поддерживаемая форма конфигурации:** `board_config.h:87–89` (BOARD_MINISTEND)
  уже задаёт `DW_RX_LISTEN_DEV = DW_TX_SOURCE_DEV = DW_DEV_M1` (один модуль, оба
  направления) — compile-time.
- Раздельность = разные SPI/CS ДВУХ чипов; свести на один чип — вопрос индекса (роли),
  не железа.

## 3. Полудуплекс на одном чипе — что НЕ реализовано (ключевое)
**Кирпичи есть все:** `dwt_rxenable` (RX_START/PollRadio), `dwt_starttx` + busy-wait
`TXFRS` (TX_FRAME `:1059–1069` / PollTx), `dwt_forcetrxoff` (TX_STOP), `dwt_rxreset`
(ALL_RX_ERR).

**НЕ реализован сам ПЕРЕХОД RX→TX→RX на одном чипе:**
- `HandleTX_FRAME`/`PollTx` **НЕ вызывают `dwt_forcetrxoff` перед `dwt_starttx`** (в
  двухчиповой схеме не нужно — M1 никогда не в RX). На ОДНОМ чипе, который непрерывно
  перевзведён в RX (`PollRadio` включает `rxenable` после каждого кадра), вызвать
  `starttx` при активном RX — конфликт: EVK этого избегает, передавая СРАЗУ после RXFCG
  (RX уже простаивает), а у нас TX-команда приходит позже, когда чип снова в RX.
- После TX **нет `dwt_rxenable`** в TX-пути — перевзвод RX делает только `PollRadio`
  (по `DW_RX_LISTEN_DEV`); на двух чипах это ДРУГОЙ чип, на одном — тайминг/координация
  между PollTx и PollRadio не определены.
- **Loopback (M1 шлёт, M2 слушает) — это ДВА чипа, не переход на одном.** Кода,
  выполняющего `rxenable→forcetrxoff→starttx→wait TXFRS→rxenable` на ОДНОМ чипе, нет.

**Итого недостаёт:** оркестрации однокадрового полудуплекса на общем чипе (заглушить RX
→ передать → дождаться TXFRS → снова слушать) и координации, чтобы `PollRadio` не
перевзводил RX, пока висит TX. Отдельные операции есть — связки нет.

## 4. Как выключить второй модуль (M1), чтобы не наводил
Доступно в API (применяется per-chip: `deca_port_select_device(M1)` сначала, второй чип
не трогается — отдельные SPI/CS):
- **`dwt_forcetrxoff()`** (`api.h:711`) — transceiver в IDLE (ни TX, ни RX). Дёшево;
  снимает активное излучение, но PLL/тактовый генератор чипа могут ещё наводить.
- **`dwt_entersleep()`** (`api.h:941`) — глубокий сон (минимум наводок), но требует
  предварительного `dwt_configuresleep()` (`:927`) и пробуждения `dwt_spicswakeup()`
  (`:984`) / тоггл CS. Сложнее (конфиг сна + wake-последовательность).
- Сейчас к M1 **не применяется НИ то, ни другое** — после INIT M1 остаётся
  инициализирован и в станционном режиме АКТИВНО передаёт (`PollTx`/`TX_FRAME`) — это и
  есть самоглушение.

## 5. INIT с одним живым модулем
- `HandleINIT` (`:447–493`) обнаруживает живые по DEV_ID, `RADIO_ERROR` **только при 0
  живых**. Плата только с M2 (M1 отсутствует): `live_count=1`, `live_mask=0x02`, INIT → **OK**.
- НО TX-путь жёстко хочет `DW_TX_SOURCE_DEV=M1`: `HandleTX_FRAME`/`PollTx`/… проверяют
  `dw_dev_state[M1].initialized` → M1 нет → **`RADIO_ERROR`**. ⇒ на одномодульной (M2)
  плате приём работает, **передача — нет** (роль TX прибита к отсутствующему M1). Это и
  есть барьер совместимости с «третьей платой только M2».

## 6. Как сделать РЕЖИМОМ, не сломав двухмодульный
**Что завязано на «ровно два модуля с фиксированными ролями»:**
- `#define DW_TX_SOURCE_DEV`/`DW_RX_LISTEN_DEV` — compile-time индексы во всех точках §1
  (≈13 мест). Рантайм-переключение требует **переменной роли** (общий «активный чип» для
  TX и RX в однокабельном режиме) вместо `#define`.
- `DW_DEVICE_COUNT=2` — уже обобщено (INIT/GET_STATUS цикл по `DW_DEVICE_COUNT`).
- Двухчиповые эксперименты (CIR/метрики/loopback) требуют ДВУХ чипов → однокабельный —
  **отдельный режим**, не замена.

**Два пути (стоимости, без выбора — за архитектором):**
- **(а) Compile-time** новой сборкой/`#define` (как `BOARD_MINISTEND` уже делает один
  модуль): дёшево по коду, но переключение = пересборка/перепрошивка; двухмодульный и
  однокабельный — разные бинарники.
- **(б) Рантайм** переменной ролей (команда/поле): ~13 точек §1 заменить `#define` на
  переменную (механически, мало) + **добавить связку полудуплекса §3** (forcetrxoff→
  starttx→rxenable, координация с PollRadio) — это ОСНОВНОЙ новый код. Плюс — выключение
  неиспользуемого чипа (§4).

**Главная стоимость в обоих — не роли (их индекс рантаймовый), а отсутствующая
оркестрация RX↔TX на одном чипе (§3).**

## EVK-референс (как делает `ss_resp_main.c` — один чип, полудуплекс)
`docs\reference\decawave-examples\ss_resp_main.c`: цикл на ОДНОМ чипе —
`dwt_rxenable(DWT_START_RX_IMMEDIATE)` (`:100`) → опрос `SYS_STATUS` на `RXFCG |
ALL_RX_TO | ALL_RX_ERR` (`:103`) → на RXFCG снять флаг (`:120`) → **`dwt_starttx(
DWT_START_TX_DELAYED)`** (`:155`) → снова `rxenable`. TX идёт СРАЗУ после приёма (RX уже
простаивает), поэтому `forcetrxoff` не нужен; используется отложенный TX. Это ровно тот
переход RX→TX→RX на одном чипе, которого у нас нет (у нас два чипа с фиксированными ролями).
