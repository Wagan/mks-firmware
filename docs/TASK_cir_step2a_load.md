## 1. Цель — нащупать потолок USB CDC и разделить причины

Снять матрицу замеров на СВОЁМ управляемом источнике (loopback M1→M2, киты выключены),
меняя две оси:
- **темп источника:** период txperiodic (крупный → мелкий, до минимума прошивки);
- **размер кадра потока:** «метрики+CIR» (~168 б) vs «только метрики» (~30 б).

Логика вывода:
- Если на лёгком потоке (метрики) FPS заметно ВЫШЕ, чем на тяжёлом (CIR) →
  потолок в USB/размере кадра (CDC не успевает гнать байты).
- Если FPS одинаков на лёгком и тяжёлом → потолок в РАДИО/приёме (M2 не принимает
  чаще), USB ни при чём.
- DROPPED растёт → USB CDC BUSY (не успевает отдавать). host_lost растёт → хост
  (Python) не читает. Оба ~0, а FPS упёрся в плато → потолок в темпе TX/радио.

## 2. Прошивка — лёгкий режим потока (минимальная правка)

Расширить `SET_STREAM_MODE (0x42)`: параметр `mode u8`:
- `0` = выкл (как сейчас),
- `1` = вкл, PAYLOAD = **метрики + окно CIR** (как сейчас, поведение НЕ меняется),
- `2` = вкл, PAYLOAD = **только метрики** (30 байт, без блока CIR).

Реализация:
- Хранить текущий режим содержимого, напр. `static uint8_t stream_content;`
  (1=метрики+CIR, 2=метрики). При `mode==0` — `stream_active=0`. При `mode==1|2` —
  `stream_active=1; stream_content=mode;` сброс seq/dropped (как сейчас при включении).
  `mode>2` → INVALID_PARAM.
- В `PROTOCOL_SendStreamFrame`: если `stream_content == 2` — НЕ добавлять блок CIR
  в PAYLOAD (только 30 байт метрик). LEN16/CRC считаются от фактической длины —
  формулы уже общие, просто PAYLOAD короче. Заголовок (SMARK/LEN16/SEQ/DROPPED) и
  CRC — без изменений.
- Хост должен уметь отличить, есть ли CIR в кадре. Простой и надёжный способ:
  **добавить 1 байт `content` в заголовок потокового кадра** сразу после DROPPED
  (значение 1 или 2). Тогда хост по нему знает, парсить CIR или нет, а не гадает по
  длине. Это меняет формат кадра — допустимо (поток ещё не «боевой», проверялся один
  раз). Обновить SPEC и парсер probe соответственно.

  > Итоговый заголовок потокового кадра после этой правки:
  > `SMARK(2) | LEN16(2) | SEQ(2) | DROPPED(2) | CONTENT(1) | PAYLOAD | CRC(1)`,
  > где LEN16 покрывает SEQ+DROPPED+CONTENT+PAYLOAD (т.е. всё между LEN16 и CRC).
  > CONTENT=1 → PAYLOAD = метрики30 + окноCIR; CONTENT=2 → PAYLOAD = метрики30.

- Компиляция: Debug чистая. Проверить, что при content=1 кадр идентичен прежнему
  плюс один байт CONTENT (не забыть учесть его в LEN16 и CRC).

> Прошивку по TX-темпу НЕ трогаем: используем существующий `TX_PERIODIC` как есть
> (минимальный период `TX_PERIOD_MIN_MS`, что уже в коде — НЕ менять). Нагрузку
> задаём периодом с хоста.

## 3. Хост — расширить tools/mks_stream_probe.py (loopback-нагрузка + режим потока)

Добавить опциональные аргументы (argparse), НЕ ломая текущий запуск:
- `--txperiodic <ms>` — если задан: перед включением потока послать команду
  `TX_PERIODIC` (0x21) с этим периодом и небольшим payload (напр. `DE AD BE EF 01`),

- `--content <1|2>` — что слать в потоке: 1=метрики+CIR (дефолт), 2=только метрики.
  Передаётся в `SET_STREAM_MODE` как param mode (1 или 2).
- `--seconds <N>` — авто-стоп через N секунд (удобно для серии замеров без Ctrl+C).
  Если не задан — работать до Ctrl+C, как сейчас.

Парсер потокового кадра обновить под новый заголовок (CONTENT-байт): по CONTENT
решать, есть ли блок CIR. Метрики парсить всегда; CIR — только при CONTENT=1.

Статистику печатать как сейчас (fps, DROPPED, host_lost, crcErr) + в заголовке
прогона печатать параметры (txperiodic, content) — чтобы в отчёте было видно, при
каких условиях снята строка. В финальном итоге добавить: заявленный период TX,
режим content, средний FPS, пиковый DROPPED, host_lost.

> Важно (последовательность на старте при --txperiodic): init → SET_PHY (Mode 3) →
> rx_start → TX_PERIODIC(период) → SET_STREAM_MODE(content). Т.е. сначала поднять
> приём и запустить периодику TX, потом включить поток. При выключении: SET_STREAM_MODE 0,
> flush, TX_STOP (0x22), rx_stop. Использовать уже существующие методы mks_protocol
> (tx_periodic, tx_stop и пр.); если метода set_stream_mode с параметром content нет —
> он есть (param mode), передать 1/2.

Аддитивно в mks_protocol.py — только если чего-то не хватает (напр. параметр content
уже покрывается существующим set_stream_mode(mode)). Ничего не удалять/не переименовывать.

## 5. Проверка на железе (ВАГАН, не CC) — процедура нагрузочного замера

Предусловие: **киты EVK ВЫКЛЮЧЕНЫ** (чистый loopback M1→M2), плата МКС, внешнее
питание, порт свободен (GUI/консоль закрыты).

Серия замеров (каждый ~20–30 с, можно через `--seconds`):

A. Тяжёлый поток (метрики+CIR), нагрузка по убыванию периода:
```
python mks_stream_probe.py COM3 --content 1 --txperiodic 50 --seconds 25
python mks_stream_probe.py COM3 --content 1 --txperiodic 20 --seconds 25
python mks_stream_probe.py COM3 --content 1 --txperiodic 10 --seconds 25
python mks_stream_probe.py COM3 --content 1 --txperiodic 5  --seconds 25
```
B. Лёгкий поток (только метрики), те же периоды:
```
python mks_stream_probe.py COM3 --content 2 --txperiodic 50 --seconds 25
python mks_stream_probe.py COM3 --content 2 --txperiodic 20 --seconds 25
python mks_stream_probe.py COM3 --content 2 --txperiodic 10 --seconds 25
python mks_stream_probe.py COM3 --content 2 --txperiodic 5  --seconds 25
```

Для каждого прогона записать в отчёт: период TX, content, средний FPS, DROPPED (fw),
host_lost, crcErr.

Что ищем:
- На каком периоде FPS выходит на ПЛАТО (перестаёт расти при уменьшении периода) —
  это потолок текущей конфигурации.
- Отличается ли плато для content=1 (CIR) и content=2 (метрики):
  - content=2 заметно выше → упираемся в USB/размер кадра;
  - одинаково → упираемся в радио/приём M2 (Mode 3, 110k — кадр долго летит).
- Появляется ли DROPPED (fw) при мелком периоде → USB CDC BUSY (потолок отдачи).
- host_lost при высоком FPS → хост (Python) не успевает читать.

Опционально (если интересно и время есть): повторить пару точек в Mode 4 (6.8 Mbps,
короткая преамбула) — там радио-кадр короче, приём быстрее; поможет отделить радио
от USB. Но это НЕ обязательно для первого нагрузочного замера.
