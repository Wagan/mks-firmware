# RECON — приёмный тракт: RX-регистры и стабильность (§15.1 дальность)

**Тип:** разведка по коду (правки НЕ вносились). **Дата:** 2026-07-22.
**Повод:** связь неустойчива на ВСЕХ уровнях TX (вплоть до железного макса), рвётся
случайно даже на 1–2 м без препятствия; антенны штатные EVK, тракт исправен. Гипотеза —
приёмная настройка прошивки. **Источник:** `App\protocol\protocol.c`,
`Drivers\decadriver\deca_device.c`, `deca_device_api.h`, референс EVK
(`docs\reference\decawave-examples\ss_*_main.c`). **Сверено с UM §2.5.5.** Только факты.

---

## ГЛАВНЫЙ ВЫВОД (гипотеза RX-регистров ОПРОВЕРГНУТА кодом)
**Мы используем штатный `dwt_configure()`, и он выставляет ВСЕ приёмные регистры из
UM §2.5.5 сам — под наш Mode 3 (ch2, PRF64, 110k, преамбула 1024, PAC32).** Ни один
RX-регистр НЕ «выпал» (в отличие от TX_POWER, который был вне `dwt_configure`). Наш
`dwt_config_t` корректен для Mode 3. ⇒ **причина нестабильности НЕ в RX-регистрах.**
Реальное отличие от EVK — в **обслуживании приёма** (§3): приёмник выключен на время
покадрового чтения CIR-аккумулятора + USB-стрима.

## 1. Что кладём в `dwt_config_t` (`HandleSET_PHY_CONFIG`, `protocol.c:653–669`) — Mode 3 ✅
`chan=2` (map_channel), `dataRate=DWT_BR_110K` (map_datarate 0), `txPreambLength=
DWT_PLEN_1024` (map_plen 1024), `prf=DWT_PRF_64M` (map_prf 64), `rxPAC=DWT_PAC32`
(map_pac 32), `txCode=rxCode=9`, `nsSFD=1` (110k), `phrMode=DWT_PHRMODE_STD`, `sfdTO=0`
→ драйверный `DWT_SFDTOC_DEF`. **Все поля под Mode 3 — верны.** LDE-микрокод загружен:
`dwt_initialise(DWT_LOADUCODE)` (`protocol.c:466`).

## 4. РЕАЛЬНОЕ отличие от EVK — обслуживание приёма (кандидат №1 нестабильности)

`PROTOCOL_PollRadio`, ветка RXFCG (`protocol.c:1315–1333`): порядок на КАЖДЫЙ принятый кадр:
```
readrxdata -> readdiagnostics -> rxpacc_nosat
 -> PROTOCOL_CaptureCIR()      (:1327)  ЧТЕНИЕ CIR-аккумулятора ~250 Б по SPI
 -> if(stream) SendStreamFrame (:1330)  USB-передача ~168 Б (может блокироваться/дропать)
 -> dwt_rxenable()             (:1333)  RX включается ПОСЛЕДНИМ
```
- **Приёмник ВЫКЛЮЧЕН всё это время** (от RXFCG до `dwt_rxenable`). Любой кадр,
  пришедший в это окно, **теряется** — независимо от уровня сигнала. Это объясняет
  «рвётся случайно даже на 1–2 м на всех уровнях мощности».
- **`PROTOCOL_CaptureCIR()` вызывается БЕЗУСЛОВНО** (`:1327`) — даже когда CIR не нужен
  (станция работает на `content=4` = метрики+данные, БЕЗ окна CIR; content=2/3 тоже без
  CIR). Т.е. каждый кадр тратит ~250 Б SPI-чтения аккумулятора **впустую**, удлиняя
  окно выключенного RX.
- **Комментарий `:1261`** объясняет, ПОЧЕМУ CIR читается до перевзвода: accumulator
  затирается при `dwt_rxenable`. Отсюда тупик: чтобы снять CIR — RX должен быть выключен;
  но выключенный RX теряет кадры. Для станции (CIR не нужен) это чистые потери.
- **Одиночный RX-буфер** (§3): пока обслуживаем кадр N, кадр N+1 не буферизуется → теряется.
- **EVK/референс** (`ss_*_main.c`): CIR-аккумулятор покадрово НЕ читает, по USB не
  стримит, обслуживание в прерывании короткое, RX перевзводится быстро. Отсюда его
  стабильность.
- Приём у нас на **polling SYS_STATUS** (не EXTI) — сам по себе не роняет кадры (RX
  остаётся включён до прихода кадра), но задерживает реакцию, если main loop занят.

## 5. Расхождения с UM/EVK (сводка)
- **RX-регистры §2.5.5** — расхождений НЕТ, `dwt_configure` ставит всё под Mode 3.
- **Антенная задержка** — не выставлена (некритично для связи).
- **Обслуживание RX** — расходится с EVK: RX выключен на время покадрового чтения
  CIR-аккумулятора (безусловного, даже когда CIR не нужен) + USB-стрима; одиночный
  буфер. **Это, а не тюнинг регистров, — вероятная причина случайных разрывов.**

---

## 6. Вывод
Гипотеза «недонастроен приёмный регистр (как выпал TX_POWER)» — **опровергнута**:
`dwt_configure()` выставляет весь §2.5.5 корректно под Mode 3. Нестабильность вероятнее
всего от **окна выключенного RX** на каждый кадр (чтение CIR-аккумулятора + USB-стрим до
`dwt_rxenable`), усугублённого **безусловным `CaptureCIR` даже без надобности** (station
= content=4) и **одиночным буфером**. Возможные направления (решение за архитектором,
правок не вносил): не читать CIR, когда он не нужен (content 2/3/4); перевзводить RX
раньше тяжёлой работы; двойной буфер / EXTI; вынести CIR-чтение из горячего пути. Плюс —
при желании — калибровка антенной задержки (для дальнометрии, не для этой проблемы).
