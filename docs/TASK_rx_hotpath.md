# RX горячий путь: минимизация окна выключенного RX (§15.1 дальность)

**Дата:** 2026-07-22. **Основание:** RECON_rx_tuning.md (`8d64f4d`) — RX-регистры верны
(dwt_configure), реальная причина потери кадров: приёмник выключен на каждом кадре на
время `CaptureCIR` (~250 Б SPI, безусловно) + USB-стрима, `dwt_rxenable` последним →
кадры в окне теряются на любой мощности. EVK-киты на том же столе бьют 5–8 м.
PLAN согласован (`docs\PLAN_rx_hotpath.md`, ORDER: A + B). Правка прошивки, горячий путь.

## Рамка/ORDER (утв.)
- **A:** `need_cir = (stream_content==1) || (!stream_active)` — снимать CIR при потоке
  content=1 ИЛИ в командном режиме (для GET_CIR); пропускать только при активном потоке
  content=2/3/4 (станция).
- **B:** единый `dwt_rxenable` + условный `CaptureCIR` (по §3 PLAN).
- content=1 CIR-путь сохранить; GET_CIR-команду, ALL_RX_ERR-ветку, регистровый тюнинг
  RX, антенную задержку — не трогать.

## Шаг 1 — прошивка (`App\protocol\protocol.c`, `PROTOCOL_PollRadio` RXFCG, коммит)
Было (окно выключенного RX = CaptureCIR + USB-стрим):
```c
PROTOCOL_CaptureCIR();                       // безусловно, ~250 Б SPI
if (stream_active) PROTOCOL_SendStreamFrame(); // USB
dwt_rxenable(DWT_START_RX_IMMEDIATE);        // RX последним
```
Стало:
```c
int need_cir = (stream_content == 1) || (!stream_active);
if (need_cir) PROTOCOL_CaptureCIR();         // снимок (копия) ДО rxenable — только когда нужен
dwt_rxenable(DWT_START_RX_IMMEDIATE);        // RX включаем РАНЬШЕ USB-стрима
if (stream_active) PROTOCOL_SendStreamFrame(); // USB — приёмник уже слушает
```
- Кадр/диагностика (`readrxdata`/`readdiagnostics`/`rxpacc_nosat`) читаются в RAM ДО
  этого блока — ранний rxenable их не теряет.
- **content=2/3/4 (станция):** `need_cir=0` → CaptureCIR пропущен, rxenable сразу.
  Из горячего пути убрано ~250 Б SPI-чтения аккумулятора.
- **content=1:** `need_cir=1` → CaptureCIR (копия в `cir_snap`) → rxenable →
  SendStreamFrame читает копию. `dwt_rxenable` затирает *живой* accumulator (регистр),
  но не `cir_snap` (RAM) — регресса нет.
- **Командный режим (стрим выкл):** `need_cir=1` (`!stream_active`) → CaptureCIR
  выполняется → **`GET_CIR` (консольная `cir`) работает** как прежде.
- Метки: RXFCG-правку нашли/тестировали **Wagan/Andrey**; настройку TX довёл **Dima**
  (пометка у TX-калибровки). Синтакс-чек `-Wall -Wextra` — чисто.

## Шаг 2 — доки (этот коммит)
- `PROTOCOL_SPEC.md` §8 (поток): заметка «Горячий путь приёма» — RX перевзводится до
  USB-стрима; CIR читается только при content=1 или в командном режиме.
- `PROJECT_HANDOFF.md` §15.1: «Правка 3 (горячий путь приёма — реальная причина)».
- Этот отчёт.

## Что изменилось по сути
- **До:** на каждый принятый кадр приёмник выключен на `CaptureCIR`(~250 Б)+USB → в это
  окно входящие кадры теряются (независимо от мощности) → случайные разрывы на 1–2 м.
- **После:** приёмник включается сразу после чтения кадра в RAM; тяжёлая работа (USB)
  идёт при уже слушающем RX; лишнее чтение CIR у станции убрано. Окно потерь схлопнуто.

## Риски (проверено)
- **GET_CIR** — сохранён (условие `!stream_active` покрывает командный режим). ✅
- **content=1 CIR** — сохранён (копия `cir_snap` до rxenable). ✅
- **Ранний rxenable + одиночный буфер** — новый кадр во время USB буферизуется железом,
  наши RAM-копии сняты → текущий стрим-кадр корректен; новый разберётся следующим
  `PollRadio`. ✅

## Приёмка на железе (ВАГАН) — замер дальности
1. Прошить. Две станции/GUI (Mode 3), «Стандарт (под маску)».
2. **Замер устойчивой дальности** — ожидание: скачок к EVK-подобной **5–8 м** (окно
   потери кадров схлопнуто; проверка «человек между станциями» — рвётся ли ещё).
3. Сравнить RSSI/стабильность до/после на фикс. дистанции.
4. Регресс: `content=1` (поток метрики+CIR, GUI/probe) — CIR/водопад целы; консольная
   `cir` (GET_CIR в командном режиме) — работает; чат/приём станции — целы.

**Отчёт Вагана:** устойчивая дальность до/после; рвётся ли ещё на близи; регресс
content=1 CIR / GET_CIR / чат.

## Дальше (§15.1)
Если дальность вышла к EVK-уровню — вернуться к этапу (2) надёжного канала (ARQ). Если
всё ещё есть остаточные потери — следующие кандидаты (отдельно): двойной RX-буфер, EXTI
вместо polling, вынос диагностики из горячего пути.
