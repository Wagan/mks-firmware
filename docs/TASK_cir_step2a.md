## 1. Цель шага CIR-2a (минимальный рабочий поток + замер)

Сделать так, чтобы по команде плата САМА, без запросов, слала поток кадров данных
(метрики + окно CIR) после каждого принятого UWB-кадра. Хост только читает.

Первый заход — ПРОСТОЙ, чтобы ЗАМЕРИТЬ реальную пропускную (FPS) и потери:
- содержимое потокового кадра: метрики + **окно CIR вокруг FP** (как в командном GET_CIR);
- команда: `SET_STREAM_MODE (0x42)`, параметр `mode u8` (0=выкл/командный, 1=вкл);
- дефолт после INIT — выкл (командный режим, безопаснее);
- CRC — на каждый потоковый кадр (как в командном; проще/надёжнее);
- при занятом USB (CDC BUSY) потоковый кадр **ДРОПАЕТСЯ**, счётчик дропов ведётся;
- в заголовке кадра — `sequence` (номер кадра) и `dropped` (счётчик дропов прошивки),
  чтобы хост видел И потери по BUSY (dropped), И потери на своей стороне (дырки в sequence).

Приёмник потока в этом заходе — ОТДЕЛЬНЫЙ минимальный скрипт (не GUI): подключиться,
включить поток, читать/парсить/считать FPS и потери, печатать статистику. Задача —
понять, тянет ли CDC. Интеграция в GUI и полный CIR — БУДУЩИЕ подшаги (§8).

## 2. Факты из кода (использовать, не искать заново)

- Командный кадр (НЕ трогаем): `SYNC(0xAA 0x55) | LEN(1 байт) | CMD | PARAMS | CRC`,
  CRC8 poly=0x07 по LEN+CMD+PARAMS. Функция `PROTOCOL_CalculateCRC(data, len)` уже есть.
- Отправка в USB: `CDC_Transmit_FS(buf, len)` — **возвращает статус**; в текущем
  `PROTOCOL_SendResponse` возврат НЕ проверяется (на редких ответах не мешало).
  Для потока возврат ПРОВЕРЯТЬ: если это `USBD_BUSY` (или не `USBD_OK`) → кадр не
  ушёл → инкремент `dropped`, кадр НЕ повторять (дропаем). Точное имя константы
  (USBD_OK/USBD_BUSY) взять из вашего `usbd_cdc_if.h`/`usbd_def.h` — НЕ по памяти.
- Снимок CIR (после CIR-1, коммит 0e1787b) уже есть: `cir_snap{valid, fp_index,
  start_index, count, data[...]}`, заполняется `PROTOCOL_CaptureCIR()` в ветке RXFCG
  ДО `dwt_rxenable`. **Переиспользуем его** — CIR уже под рукой, повторно accumulator
  не читаем.
- Метрики уже считаются в `HandleGET_SIGNAL_METRICS` (30-байтный формат): count,
  maxGrowthCIR, rxPreamCount, stdNoise, firstPathAmp1..3, firstPath, maxNoise,
  rxpacc_nosat, N_corrected, RSSI(i16 dBm×100), FP_POWER(i16), A, SNR(i16 dB×100).
  Для потока переиспользовать ТУ ЖЕ логику расчёта (вынести общую сборку 30 байт в
  функцию, чтобы командный ответ и поток не расходились — см. §3.3).
- Ветка RXFCG в `PROTOCOL_PollRadio` (main loop): снимает флаг, читает длину/данные,
  `dwt_readdiagnostics`, `rxpacc_nosat`, `rx_metrics.valid=1`, `count++`,
  `PROTOCOL_CaptureCIR()`, затем `dwt_rxenable`. Поток встраиваем ЗДЕСЬ (§3.4).
- Свободные CMD-коды: 0x42 свободен (рядом с диагностикой 0x40/0x41). Берём его
  под `SET_STREAM_MODE`.

### 3.2. Потоковый кадровый формат (СВОЙ, отдельный от командного)

Определить в protocol.c (константы + сборка). Формат на проводе:
```
+--------+--------+----------+---------+----------------+-----+
| SMARK  | LEN16  | SEQ      | DROPPED | PAYLOAD        | CRC |
| 2 байта| 2 байта| 2 байта  | 2 байта | LEN16-6 байт   |1 б. |
+--------+--------+----------+---------+----------------+-----+
```
- `SMARK` — магический маркер начала потокового кадра, ОТЛИЧНЫЙ от командного SYNC
  (0xAA 0x55). Взять, например, `0xDE 0xCA` (2 байта). Это позволяет хосту отличать
  потоковые кадры и ре-синхронизироваться при сбое.
- `LEN16` — длина (SEQ+DROPPED+PAYLOAD+... ), u16 LE. 2 байта, т.к. окно CIR не
  влезает в 1 байт. Точное определение, ЧТО именно покрывает LEN16 — задать явно в
  коде и в SPEC (рекомендую: число байт ПОСЛЕ LEN16 и ДО CRC, т.е. SEQ+DROPPED+PAYLOAD).
- `SEQ` u16 LE — номер потокового кадра (инкремент на каждый ОТПРАВЛЕННЫЙ кадр;
  на дропнутый — НЕ инкрементировать SEQ, чтобы хост по дыркам ловил только СВОИ
  потери, а прошивочные видел через DROPPED; см. примечание ниже).
- `DROPPED` u16 LE — счётчик потоковых кадров, дропнутых прошивкой по BUSY (нарастающий,
  с момента включения потока). u16 может переполниться при долгом потоке — для замера
  ок; в SPEC отметить, что счётчик 16-битный (оборачивается).
- `PAYLOAD` — тело: **метрики (30 байт, тот же формат, что GET_SIGNAL_METRICS) +
  окно CIR** (заголовок окна 6 байт: fp_index u16, start_index u16, cir_count u16,
  затем cir_count пар I/Q int16 LE — тот же формат, что DATA у GET_CIR). Т.е. PAYLOAD
  = [30 байт метрик][6 байт заголовка CIR][cir_count*4 байт I/Q].
- `CRC` — CRC8 (та же функция/полином) по LEN16+SEQ+DROPPED+PAYLOAD (т.е. по всему,
  что после SMARK и до CRC). Маркер SMARK в CRC НЕ входит (как SYNC в командном).

> Примечание по SEQ/DROPPED (важно для однозначной диагностики на хосте):
> - SEQ инкрементируется только когда кадр РЕАЛЬНО отправлен (CDC вернул OK).
> - При BUSY: SEQ не трогаем, DROPPED++. Значит: хост, видя непрерывный SEQ без
>   дырок, но растущий DROPPED, понимает «прошивка что-то дропнула по BUSY».
>   Дырки в SEQ на хосте означают «кадр ушёл из прошивки, но потерялся на пути/
>   на хосте» (разрыв USB, переполнение буфера хоста). Две разные потери — раздельно.

Размерная прикидка (для буфера): при окне ~33 отсчёта (дефолт half=16) PAYLOAD ≈
30 + 6 + 33*4 = 168 байт; полный потоковый кадр ≈ 2+2+2+2+168+1 = 177 байт. При
half=30 (61 отсчёт) ≈ 2+2+2+2+(30+6+244)+1 = 289 байт. Буфер потокового кадра
заложить с запасом (напр. 320 байт). Полуширину окна для потока взять
ФИКСИРОВАННОЙ = тот же дефолт, что у командного CIR (напр. 16). Параметризацию
полуширины/содержимого — НЕ здесь (будущий подшаг).

### 3.4. Состояние потока + отправка в PollRadio

Состояние (рядом с rx_metrics/cir_snap):
```c
static volatile uint8_t stream_active;   /* 1 = потоковый режим включён */
static uint16_t stream_seq;              /* номер отправленного потокового кадра */
static uint16_t stream_dropped;          /* дропнуто по BUSY (нарастающий, u16) */
```
В `PROTOCOL_Init`: обнулить все три.

Обработчик команды:
```c
/* SET_STREAM_MODE (0x42): mode u8 (0=выкл/командный, 1=вкл поток). Сбрасывает
 * счётчики seq/dropped при ВКЛючении. DATA нет. */
static ResponseStatus HandleSET_STREAM_MODE(const uint8_t* params, uint8_t params_len,
                                            uint8_t** out_data, uint8_t* out_len)
{
    (void)out_data; *out_len = 0;
    if (params_len < 1) return STATUS_INVALID_PARAM;
    uint8_t mode = params[0];
    if (mode > 1) return STATUS_INVALID_PARAM;
    if (mode == 1) { stream_seq = 0; stream_dropped = 0; }
    stream_active = mode;
    return STATUS_OK;
}
```
Зарегистрировать в `PROTOCOL_RegisterAllHandlers`:
```c
    PROTOCOL_RegisterHandler(CMD_SET_STREAM_MODE, HandleSET_STREAM_MODE);
```

Отправка потокового кадра — в `PROTOCOL_PollRadio`, ветка RXFCG, ПОСЛЕ
`PROTOCOL_CaptureCIR()` и ДО `dwt_rxenable` (данные уже собраны; шлём, потом слушаем):
```c
    PROTOCOL_CaptureCIR();

    if (stream_active) {
        PROTOCOL_SendStreamFrame();   /* NEW: собрать и отправить потоковый кадр */
    }

    dwt_rxenable(DWT_START_RX_IMMEDIATE);
```

Функция отправки (static). Собирает кадр в статический буфер, шлёт `CDC_Transmit_FS`,
ПРОВЕРЯЕТ возврат:
```c
/* Собрать потоковый кадр (метрики + окно CIR из cir_snap) и отправить в USB CDC.
 * При BUSY (CDC не готов) — дроп: dropped++, SEQ не инкрементируем. Устройство
 * DW_RX_LISTEN_DEV уже выбрано вызывающим (PollRadio). */
static void PROTOCOL_SendStreamFrame(void)
{
    static uint8_t frame[320];  /* с запасом; см. прикидку §3.2 */
    uint8_t* p = frame;

    /* SMARK */
    *p++ = 0xDE; *p++ = 0xCA;

    /* место под LEN16 (заполним в конце) */
    uint8_t* len_field = p; p += 2;

    uint8_t* body_start = p;    /* с этого места считаем LEN16 и CRC */

    /* SEQ, DROPPED */
    PUT_U16LE(p, stream_seq);     p += 2;
    PUT_U16LE(p, stream_dropped); p += 2;

    /* PAYLOAD: 30 байт метрик */
    p += build_metrics_block(p);

    /* PAYLOAD: окно CIR из cir_snap (заголовок 6 байт + I/Q).
     * Формат идентичен DATA у GET_CIR. Полуширину НЕ применяем — берём весь снимок
     * (он и так усечён до дефолтного окна при захвате). */
    PUT_U16LE(p, cir_snap.fp_index);    p += 2;
    PUT_U16LE(p, cir_snap.start_index); p += 2;
    PUT_U16LE(p, cir_snap.count);       p += 2;
    memcpy(p, cir_snap.data, (size_t)(cir_snap.count * 4u));
    p += cir_snap.count * 4u;

    /* LEN16 = число байт от body_start до конца PAYLOAD (SEQ+DROPPED+PAYLOAD) */
    uint16_t body_len = (uint16_t)(p - body_start);
    PUT_U16LE(len_field, body_len);

    /* CRC8 по LEN16+SEQ+DROPPED+PAYLOAD = [len_field .. p-1] */
    uint8_t crc = PROTOCOL_CalculateCRC(len_field, (uint8_t)((p - len_field) & 0xFF));
    /* ВНИМАНИЕ: (p-len_field) может быть > 255 при широком окне — CalculateCRC берёт
     * uint8_t len. Для потока с окном до ~289 байт это ПЕРЕПОЛНИТ uint8_t. РЕШЕНИЕ:
     * сделать локальный расчёт CRC по uint16-длине здесь (не менять командную
     * PROTOCOL_CalculateCRC). См. §3.5. */
    *p++ = crc;

    uint16_t total = (uint16_t)(p - frame);

    /* Отправка. Возврат ПРОВЕРЯЕМ: BUSY → дроп. */
    if (CDC_Transmit_FS(frame, total) == USBD_OK) {
        stream_seq++;
    } else {
        stream_dropped++;
    }
}
```

### 3.5. CRC по длине >255 (важно — не сломать командный CRC)

Командная `PROTOCOL_CalculateCRC(const uint8_t*, uint8_t len)` берёт длину uint8_t —
для потокового кадра (тело может быть >255 байт при широком окне) этого мало. НЕ
менять командную функцию. Сделать ОТДЕЛЬНУЮ для потока:
```c
/* CRC8 (тот же полином 0x07) по буферу длиной uint16 — для потокового кадра. */
static uint8_t PROTOCOL_CalculateCRC16len(const uint8_t* data, uint16_t len)
{
    uint8_t crc = 0;
    for (uint16_t i = 0; i < len; i++) {
        crc ^= data[i];
        for (uint8_t j = 0; j < 8; j++)
            crc = (crc & 0x80) ? (uint8_t)((crc << 1) ^ 0x07) : (uint8_t)(crc << 1);
    }
    return crc;
}
```
В `PROTOCOL_SendStreamFrame` использовать её:
`uint8_t crc = PROTOCOL_CalculateCRC16len(len_field, (uint16_t)(p - len_field));`
(этот вызов ставить ДО `*p++ = crc;` — т.е. считать CRC по [len_field .. текущий p),
где p ещё указывает на место CRC.)

## 4. Хост — минимальный приёмник потока (НОВЫЙ файл tools/mks_stream_probe.py)

Отдельный маленький скрипт (не GUI, не консоль). Задача: включить поток, читать
потоковые кадры, считать статистику, печатать. Использует mks_protocol.py для
подключения/команды SET_STREAM_MODE, а разбор потокового кадра — свой (формат §3.2).

Функциональность:
1. Открыть порт (аргумент, напр. `python mks_stream_probe.py COM3`).
2. Команды через mks_protocol: `init`, `SET_PHY_CONFIG` (Mode 3 пресетом — как в
   консоли/GUI), `rx_start`. Затем `SET_STREAM_MODE 1` (послать 0x42 с param [1]).
3. Перейти в режим ЧТЕНИЯ потока: искать SMARK (0xDE 0xCA), читать LEN16, тело, CRC;
   проверять CRC; парсить SEQ, DROPPED, метрики (30 б), окно CIR.
4. Считать и раз в ~1 с печатать статистику:
   - принято кадров, кадров/с (по времени ПК),
   - последний SEQ, последний DROPPED (дропнуто прошивкой),
   - потеряно на хосте = (макс SEQ − мин SEQ + 1) − принято_кадров (дырки в SEQ),
   - последние метрики (SNR/RSSI/FP_POWER), fp_index из CIR.
5. По Ctrl+C: послать `SET_STREAM_MODE 0` (выключить поток), `rx_stop`, закрыть порт,
   напечатать итог (всего принято, средний FPS, всего дропнуто прошивкой, потеряно
   на хосте).

Требование по чтению: читать неблокирующе/порциями (как в mks_protocol MKS._read_exact
с дедлайном), чтобы Ctrl+C ловился. Ре-синхронизация: если CRC не сошёлся или
SMARK не найден — искать следующий SMARK, не падать.

Аддитивно в mks_protocol.py (если нужно): `CMD_SET_STREAM_MODE = 0x42` и метод
`set_stream_mode(mode)`. Ничего существующего не менять.

> Замечание: пока поток включён, командный канал занят потоковыми кадрами. Скрипт
> в режиме чтения потока команды НЕ шлёт (кроме финального выключения по Ctrl+C —
> его можно послать, т.к. после stream=0 прошивка перестанет слать поток; но проще:
> сначала стоп-команда, потом дочитать/сбросить буфер). CC: реализовать аккуратно —
> послать SET_STREAM_MODE 0, затем flush_input, затем rx_stop.

## 6. Проверка на железе (ВАГАН, не CC) — процедура

Предусловие: плата МКС, внешнее питание, источник Mode 3 (киты EVK в эфире или
loopback M1→M2). Командный GUI/консоль закрыты (порт занимает probe-скрипт).

1. `python mks_stream_probe.py COM3`
2. Скрипт сам: init → Mode 3 → rx_start → stream on. Пойдёт поток.
3. Смотреть статистику (раз в 1 с): кадров/с, DROPPED (прошивка), потеряно на хосте.
4. Дать поработать ~30–60 с. Ctrl+C — итог.

Что записать в отчёт (это ЦЕЛЬ шага — замер):
- Реальный FPS потока (кадров/с) при окне ~33 отсчёта.
- DROPPED прошивкой: часто ли CDC BUSY (много дропов = CDC на пределе → повод к bulk).
- Потери на хосте (дырки SEQ): успевает ли Python читать без потерь.
- Стабильность: не рассинхронизируется ли поток (CRC-ошибки, поиск SMARK).
- Сравнить FPS потока с прежним командным (~9 кадр/с в GUI) — вырос ли (должен: нет
  запрос-ответ круга).
По результату решим CIR-2b: полный CIR потоком / bulk / интеграция в GUI.
