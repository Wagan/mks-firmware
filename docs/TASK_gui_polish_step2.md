## 1. Что делаем в Шаге 2

### 1.1. Вкладки (ttk.Notebook)
Разбить окно на вкладки:
- **«Монитор»** (главная): график CIR (matplotlib), кружок-индикатор справа вверху,
  компактная телеметрия, две кнопки **Старт**/**Стоп**, блок записи (Запись/Стоп
  записи + счётчик + текущий файл). Всё для наблюдения — тут.
- **«Настройки»**: подключение (COM-порт, Подключить/Отключить), выбор PHY-режима
  (Mode 1–8 из PHY_MODES + «Ручной» 6 полей), сценарий старта (см. 1.3), content
  (лёгкий/полный, см. 1.4), настройки записи (префикс, папка, Лёгкий/Полный).
- **«Водопад»** (ЗАГЛУШКА): пустая вкладка с надписью «Водопад — в разработке (Шаг 3)».
  Каркас/переключение готово, рендер — следующий шаг. Буфер CIR (deque) уже копится
  из Шага 1 — Шаг 3 его отрисует.

> Notebook — основа и для будущего переключения CIR↔Водопад. Вкладки именно поэтому
> (не попап): под водопад нужен механизм смены экранов.

### 1.2. Две кнопки Старт/Стоп (на «Мониторе»)
- **Старт**: выполняет СЦЕНАРИЙ старта (по умолчанию: INIT → SET_PHY(выбранный режим)
  → RX_START → SET_STREAM_MODE(content)) в фоновом воркере (не морозить UI), затем
  переходит в чтение потока (движок Шага 1).
- **Стоп**: SET_STREAM_MODE 0 → flush → RX_STOP; остановить чтение.
- Отдельные кнопки INIT / «Применить режим» с главного экрана УБРАТЬ — их роль берёт
  Старт (через сценарий) и вкладка Настройки. Во время активного потока конфиг на
  вкладке Настройки блокировать (менять между запусками).

### 1.3. Сохраняемые сценарии старта (тип а1 — последовательность запуска)
Сценарий = что делает «Старт». Простой человекочитаемый формат — ТЕКСТОВЫЙ файл,
одна команда на строку, `#` — комментарий. Язык команд — как в консоли, например:
```
# сценарий старта: Mode 4, полный поток
init
setphy 2 2 128 9 64 8      # Mode 4
rxstart
stream 1                    # SET_STREAM_MODE content=1
```
UI на вкладке Настройки:
- Поле/выпадающий список активного сценария + кнопки **Загрузить…**, **Сохранить…**,
  **Сохранить как…** (диалоги файлов). Плюс кнопка **Сбросить к умолчанию** (вернуть
  дефолтную цепочку INIT→SET_PHY→RX_START→stream по текущему выбору режима/content).
- Небольшое текстовое поле (Text) для просмотра/редактирования строк сценария.
- При «Старт» — выполнять команды сценария ПОСЛЕДОВАТЕЛЬНО, показывая статус каждой
  (лог в статус-строке); если команда вернула не-OK — остановиться, показать ошибку,
  не продолжать (не уходить в поток с полу-настроенной платой).

Реализация выполнения сценария: маппинг строк на методы mks_protocol/команды —
переиспользовать то, что уже умеет консоль/GUI (init, setphy из PHY_MODES или явные
параметры, rxstart, stream N). Набор поддерживаемых в сценарии команд:
`init`, `setphy <ch> <dr> <plen> <code> <prf> <pac>` (или `mode <N>` как алиас
пресета), `rxstart`, `rxstop`, `stream <0|1|2>`, `txstop`. Неизвестная команда в
сценарии → показать ошибку, не выполнять. (Пусть будет минимальный безопасный набор,
не произвольный обработчик всего протокола.)

> Дефолтный сценарий генерится из выбранного режима + content, если пользователь не
> загрузил свой. Т.е. простой пользователь просто выбирает Mode в списке и жмёт Старт;
> продвинутый — правит/сохраняет сценарий.

### 1.4. Переключатель content (лёгкий/полный)
- На вкладке Настройки: радио/чекбокс **content**: 1 = метрики+CIR (полный),
  2 = только метрики (лёгкий). По умолчанию 1.
- Влияет на команду `stream` в сценарии старта (stream 1 или stream 2).
- **Поведение при content=2 (нет CIR):** график CIR на «Мониторе» и вкладка «Водопад»
  показывают заглушку «CIR отключён (content=2 — только метрики)». Кружок-индикатор,
  телеметрия, запись метрик — работают. Т.е. content=2 = режим «только метрики»,
  CIR-графика недоступна (честно, без попыток рисовать пустоту).
- Запись при content=2 — только metrics.csv (cir.csv не пишется, т.к. CIR нет).

### 1.5. Сохранить рабочим весь движок Шага 1
- Поток, StreamReader, кружок (зелёный/серый), гашение по таймауту (>1с → серый +
  чистка графика), прорежённый рендер (~16/с) при записи ВСЕХ кадров, буфер водопада
  (deque 500), запись CSV из потока (формат прежний), три страховки от залипшего
  потока (при подключении flush+stream0; при стопе/закрытии stream0; прошивочный INIT).
- Телеметрия (fps/count/SNR/RSSI/FP_POWER/FP_INDEX/DROPPED/host_lost) — на «Мониторе»,
  компактно.

## 4. Проверка на железе (ВАГАН)
Передатчик (плата 2, Mode 4, txperiodic); приёмник (COM3).
1. GUI → вкладка Настройки: Подключить (COM3), выбрать Mode 4, content=1. (Сценарий
   по умолчанию сгенерён.)
2. Вкладка Монитор → Старт → сценарий выполняется (в статусе видно init/setphy/
   rxstart/stream OK) → кружок зелёный, CIR живой, телеметрия fps~200.
3. Стоп → поток остановлен. Выключить TX → кружок серый/график гаснет (если был Старт).
4. Сценарий: на Настройках отредактировать (например сменить на Mode 3), Сохранить
   как файл, Загрузить обратно, Старт → применился новый режим.
5. content=2 → Старт → кружок зелёный, телеметрия идёт, график CIR — заглушка
   «CIR отключён»; запись → только metrics.csv.
6. content=1 → запись Полный → metrics.csv + cir.csv (все кадры, frame_id).
7. Закрыть GUI / убить GUI → консоль init OK (страховки целы).
8. Вкладка Водопад — заглушка (рендер в Шаге 3), буфер копится (проверим в Шаге 3).
