# Канал данных поверх UWB — §15.1 стадия (0): фундамент

**Дата:** 2026-07-22. **Основание:** RECON_data_channel_stage0.md (5d3f45a) + рамка
архитектора. **Цель стадии:** отдать тело принятого UWB-кадра наружу на хост через
новый потоковый режим `content=3` (фундамент для чата/надёжного канала/звука).

Разбито на **два малых шага** (прошивка отдельно от хоста), каждый со своей проверкой.

---

## Рамка архитектора (соблюдена)
- Новый `content=3` в `SET_STREAM_MODE`; несёт **только данные** — ни метрик, ни CIR.
- PAYLOAD при `content=3`: `data_len (u16 LE) | data[data_len]`; `data` = тело кадра
  **без FCS** (`frame_len − 2` из `rx_frame`). При `frame_len < 2` или `> 128` — не слать.
- `content=1/2` не менять. Кадр на каждый RXFCG (без пейсинга). `frame[320]` не расширять.
- CRC8/LEN16 — как в существующем пути. Хост: разбор `content=3` в `mks_stream.py` +
  минимальный приёмник. GUI не трогать.

**Отступление (согласовать при желании):** `build_metrics_block` на прежней `:1133`
вызывался **безусловно**. Чтобы честно выполнить «ни метрик» для `content=3`, блок
метрик сделан **условным** (только `content 1/2`), а ветка данных — на его месте, а не
буквально «после CIR». Формат `data_len|data` (без метрик) выдержан точно.

## Шаг 1 — прошивка (коммит: content=3)
Файл `App/protocol/protocol.c`:
- `HandleSET_STREAM_MODE`: порог `mode > 2` → `mode > 3`; `mode=3` включает поток,
  `stream_content=3` (присваивание уже обобщённое). `mode>3` → `INVALID_PARAM`.
- `PROTOCOL_SendStreamFrame`: метрики (`build_metrics_block`) сделаны условными
  (`content 1/2`); добавлена ветка `content==3`:
  - ранний guard: `frame_len < 2 || frame_len > RX_FRAME_MAX(128)` → `return` (не шлём,
    SEQ/DROPPED не трогаем);
  - PAYLOAD = `data_len (u16 LE)` + `memcpy(rx_frame, data_len)`, где `data_len =
    frame_len − 2` (отброшен FCS);
  - LEN16/CRC8/отправка — общий хвост функции (без изменений).

**Проверка (моя):** `arm-none-eabi-gcc -fsyntax-only -Wall -Wextra` с флагами проекта
(`-DDWT_NUM_DW_DEV=2 -DBOARD_MKS ...`) — **чисто, без предупреждений**.
**Проверка на железе — за Ваганом** (см. «Приёмка» ниже).

## Шаг 2 — хост (коммит: разбор + приёмник)
- `tools/mks_protocol.py` `set_stream_mode`: допускает `mode=3` (было 0/1/2).
- `tools/mks_stream.py` `parse_stream_body`: ветка `content==3` → `data_len u16 LE` +
  `data`; в возврат добавлен ключ `data` (None для `content 1/2`); устойчивость к
  короткому/усечённому телу (`ProtocolError`, не падение). `content=1/2` не изменены.
- `tools/mks_data_probe.py` (**новый**) — минимальный приёмник: `init → set_phy(Mode) →
  rx_start → [tx_periodic] → set_stream_mode(3)`, печать принятых байт (hex + ASCII),
  сверка с `--payload`, учёт SEQ/DROPPED/host_lost, итоговый вердикт приёмки.

**Проверка (моя):** `py_compile` всех затронутых + юниты: разбор `content=3` (seq/data),
пустой payload, усечённое тело → `ProtocolError`, регресс `content=2`, `set_stream_mode(3)`
— **все прошли**.

## Формат кадра content=3 (итог, для хоста/MATLAB)
```
SMARK(0xDE 0xCA) | LEN16(u16 LE) | SEQ(u16) | DROPPED(u16) | CONTENT=3
                 | data_len(u16 LE) | data[data_len] | CRC8
```
`data` = тело принятого кадра без FCS. `LEN16` = байты `SEQ..data` (как обычно).
`CRC8` (poly 0x07) по `[LEN16 .. конец data]` (SMARK не входит). SPEC v1.5 §8 обновлён.

## Приёмка стадии (0) — на железе (ВАГАН), loopback M1→M2
Self-contained на ОДНОЙ плате МКС (M1 TX → M2 RX), COM3. Прошить свежую прошивку.

1. Запустить приёмник с известным payload и передатчиком-петлёй:
   ```
   python tools/mks_data_probe.py COM3 --mode 3 --txperiodic 50 --payload "DE AD BE EF 01"
   ```
   (Mode 3 — под слушать; для чистого loopback можно любой согласованный режим.)
2. Ожидаемо в выводе:
   - строки принятых кадров: `SEQ=… len=5  DE AD BE EF 01  |.....|  == payload OK`;
   - `data` **байт-в-байт** равен переданному payload (FCS отрезан прошивкой);
   - `SEQ` растёт, `DROPPED(fw)=0`, `host_lost=0` на умеренном темпе (50 мс);
   - в ИТОГЕ: `сверка с payload: OK=… MISMATCH=0`, `ПРИЁМКА СТАДИИ 0: ПРОЙДЕНА`.
3. Негатив-проверки (по желанию):
   - без `--txperiodic` (пассив, эфир тихий) — кадров данных нет (нечего слать) — ОК;
   - другой payload/длина — `data` следует за ним (проверить len и содержимое);
   - старая прошивка (без content=3) — `SET_STREAM_MODE 3` вернёт `INVALID_PARAM`,
     скрипт честно сообщит и остановится.
4. Регресс: `content=1/2` не затронуты — `mks_stream_probe.py --content 1/2` и GUI
   работают как прежде (метрики/CIR/водопад).

**Отчёт Вагана:** совпали ли данные байт-в-байт, рос ли SEQ, был ли DROPPED/потери,
не сломались ли content=1/2.
