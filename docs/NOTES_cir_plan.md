## 1. Целевая картина (чего хотим от CIR)

- **Консоль** (`mks_console.py`) — CIR **усечённо**: окно ±N отсчётов вокруг FP
  (first path). Для быстрой диагностики формы первого пути. Влезает в обычный
  протокольный кадр.
- **Отдельная утилита водопада** (напр. `mks_cir_waterfall.py`) — **полный CIR**
  (~1016 отсчётов I/Q), псевдографика-водопад. НЕ в консольном выводе — отдельное
  окно/вывод. Нужен на вторых шагах, не на первых.
- Полный CIR нужен обязательно (не только окно) — вопрос лишь в транспорте и FPS.

## 2. Архитектура: два режима, каждая утилита объявляет свой при старте

Идея Вагана: **два режима работы прошивки**, клиент при запуске шлёт команду
переключения.
- Консоль при старте → «консольный режим» (CIR по запросу, усечённо).
- Утилита водопада при старте → «потоковый режим» (прошивка сама шлёт CIR-кадры
  после каждого приёма).
- Новая команда переключения (напр. `SET_STREAM_MODE`, код занять в SPEC):
  0=консольный/по запросу (по умолчанию после INIT — безопаснее), 1=потоковый.

## 3. GET_CIR (0x41) — параметризованный (один код на оба клиента)

Параметры: `start_index u16, count u16`. Тогда:
- консоль просит окно (напр. FP−32 .. FP+32);
- водопад/полное чтение просит 0..1016 (чанками, если нужно).
Один обработчик покрывает оба сценария — «полный vs усечённый» решает КЛИЕНТ
параметрами, а не два разных кода.

## 4. Транспорт (ГЛАВНАЯ развязка) — НЕ ломать основной кадр

**Решение:** для потока — **отдельный кадровый формат**, НЕ раздувать основной
протокол.
- Основной кадр (SYNC|LEN|CMD|PARAMS|CRC) отлажен, на нём всё держится. Трогать
  его `LEN` ради CIR рискованно (затронет все команды, CRC, парсер).
- **Зависимость №1 для recon:** сколько байт `LEN` в текущем SPEC? Если 1 байт →
  тело ≤255, полный CIR (~4 КБ) физически не влезает в один кадр. ПРОВЕРИТЬ в
  PROTOCOL_SPEC перед любым дизайном.
- Потоковый формат — свой: расширенный заголовок (2- или 4-байтовая длина),
  магический маркер начала CIR-блока, номер кадра/отсчёта для синхронизации.
  Возможно облегчённый CRC (не на каждый отсчёт). Консольный режим остаётся на
  текущем протоколе (усечённый CIR влезает в обычный кадр).

### 4.1 Транспорт — уровень USB (обсуждено, решение отложено до замера)

- **CDC (virtual COM) сейчас.** Важно: «115200» в CDC — фикция, USB CDC игнорирует
  «бод», данные идут на скорости USB. Мы НЕ ограничены 115200.
- **Потолок железа: USB Full Speed 12 Мбит/с** (F411 имеет только FS, High Speed
  480 Мбит/с недоступен без внешнего ULPI PHY / другого чипа). Полезная пропускная
  CDC/bulk на FS реально ~0.8–1.0 МБ/с.
- **Полноценный USB (bulk) — возможен, но не панацея:** custom bulk-endpoint
  (WinUSB/libusb/pyusb) убирает CDC-оверхед, но упирается в ТОТ ЖЕ FS-потолок —
  выигрыш в эффективности, не кратный.
- **Композитное устройство CDC + bulk** (вариант «на вырост»): команды по CDC
  (консоль как есть), поток по bulk. Ложится на «два режима» идеально. Цена:
  композитный дескриптор + CubeMX USB-настройка (за Ваганом) + драйвер bulk на
  хосте (Zadig/WinUSB на Windows, pyusb вместо pyserial).
- **РЕШЕНИЕ (принцип «no guessing»): не переходить на bulk превентивно.** Сначала
  выжать CDC → ЗАМЕРИТЬ реальные FPS полного CIR → если CDC-оверхед душит, тогда
  композит CDC+bulk (апгрейд, не переделка). Не оптимизируем то, что не измерили
  как узкое место.

### 4.2 Бюджет (прикидка, не факт — только замер даст точное)

- Полный CIR = 1016 × 4 байта (I16+Q16) ≈ **4 КБ/кадр**.
- При ~0.8 МБ/с (FS) ≈ 5 мс/CIR → теоретически до ~200 CIR/с; на практике меньше
  (SPI-чтение accumulator из DW1000 тоже не мгновенно). Реалистично — десятки
  CIR/с для полного. Для водопада хватает (глазу 10–30 FPS достаточно).
- Узкие места, которые проявятся на замере: частота SPI к DW1000; реализация
  `CDC_Transmit_FS` (блокирующе / по прерыванию / двойная буферизация); успевает
  ли хост-Python (pyserial) читать без потерь (возможно, больше буфер).

## 5. Грабли: accumulator затирается при перевзводе приёма (КРИТИЧНО)

Из UM (сверено на шаге RSSI): чтение accumulator (`dwt_readaccdata`) требует
«фиктивного» первого байта (dummy), и accumulator валиден ТОЛЬКО пока приёмник не
перезапущен. **В нашем текущем PollRadio после RXFCG мы сразу `dwt_rxenable`** —
это затрёт CIR до того, как мы его вычитаем.
→ Для CIR надо будет НЕ перевзводить приём до вычитывания accumulator (или
буферизовать/читать CIR прямо в ветке RXFCG до rxenable). Аккуратно, чтобы НЕ
сломать уже работающие RSSI/FP_POWER/SNR (они читаются `dwt_readdiagnostics` в той
же ветке). Это архитектурная зависимость с PollRadio — главный риск шага 1.

## 6. Формат отсчётов CIR (проверить в recon)

- Accumulator: комплексные значения, 16-bit real + 16-bit imag на отсчёт (из
  DecaRanging Source Guide §4.5). Span: 992 (PRF16) / 1016 (PRF64) отсчётов.
- `dwt_readaccdata(buffer, len, accOffset)` — сигнатуру и dummy-байт проверить в
  vendor `deca_device_api.h` / `deca_device.c` при recon (НЕ по памяти).

## 7. Предлагаемый порядок (3 отдельных захода, каждый со своим recon)

1. **Шаг CIR-1:** `GET_CIR 0x41` параметризованный (start/count), консольное
   усечённое чтение окна вокруг FP + фикс PollRadio (accumulator не затирать).
   Проверка на железе. Транспорт — текущий CDC/протокол (окно влезает).
2. **Шаг CIR-2:** команда `SET_STREAM_MODE` + потоковый режим в прошивке + свой
   потоковый кадровый формат (§4). Замер реальных FPS/пропускной на CDC.
3. **Шаг CIR-3:** утилита-водопад (хост, Python), полный/децимированный CIR,
   псевдографика. При необходимости (по замеру шага 2) — композит CDC+bulk (§4.1).
