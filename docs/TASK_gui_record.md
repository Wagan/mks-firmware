## 1. Цель

Добавить в GUI **запись данных в CSV** во время детекции, с переключателем
**Лёгкий / Полный**. Данные берутся из ТОГО ЖЕ фонового потока опроса, что уже
кормит лампу/график (НЕ добавлять отдельный опрос порта).

- **Лёгкий**: одна строка на принятый кадр — метрики. Файл `*_metrics.csv`.
- **Полный**: метрики (как лёгкий) + форма CIR по отсчётам. Два файла:
  `*_metrics.csv` и `*_cir.csv`, связанные по `frame_id`.

Запись — **только по уникальным кадрам** (когда `count` из метрик вырос относительно
предыдущего). Это исключает дубли одного кадра при опросе чаще, чем приходят кадры.

---

## 2. Формат файлов (заложить точно так)

Разделитель — запятая. Первая строка — заголовок (имена колонок). Числа — как числа
(не строки). Невалидные значения (по флагам `*_valid` из `parse_signal_metrics`) —
писать **пустую ячейку** (пусто между запятыми), чтобы MATLAB `readmatrix`/pandas
читали их как NaN. Кодировка — UTF-8. Перевод строк — стандартный (csv-модуль сам).

### 2.1. metrics.csv (оба режима)

Колонки (в этом порядке):
```
frame_id, timestamp, count, frames_per_sec, RXPACC, RXPACC_NOSAT, N_corrected,
CIR_PWR, STD_NOISE, FP_INDEX, RSSI_dBm, FP_POWER_dBm, SNR_dB, mode
```
- `frame_id` — сквозной счётчик записанных кадров в этой сессии записи (0,1,2,…).
  НЕ равен `count` (count — счётчик прошивки с момента RX_START). frame_id —
  локальный ключ для связи с cir.csv.
- `timestamp` — время ПК на момент записи кадра. Формат ISO-8601 с миллисекундами,
  напр. `2026-07-17T20:55:06.123` (локальное время; `datetime.now().isoformat(timespec='milliseconds')`).
- `count` — из метрик (счётчик кадров прошивки).
- `frames_per_sec` — та же оценка кадров/с, что уже показывается в GUI (если считается).
- `RXPACC, RXPACC_NOSAT, N_corrected, CIR_PWR, STD_NOISE, FP_INDEX` — из
  `parse_signal_metrics` (сырые/строгие поля). FP_INDEX — сырой из метрик
  (поле `FP_INDEX`); НЕ путать с fp_index из CIR (тот в отсчётах).
- `RSSI_dBm, FP_POWER_dBm, SNR_dB` — строгие (финальный формат метрик); если
  соответствующий `*_valid == False` → пустая ячейка.
- `mode` — строка текущего применённого режима (напр. "Mode 3" или "Ручной").

> Если поля RXPACC_NOSAT / N_corrected доступны только в финальном формате метрик
> (28/30 байт) — при интерим-формате (18 байт) писать пусто. Ориентироваться на
> ключ `format` в результате `parse_signal_metrics`.

### 2.2. cir.csv (только полный режим)

Одна строка на ОДИН отсчёт CIR. Колонки:
```
frame_id, sample_index, I, Q, amplitude
```
- `frame_id` — тот же, что в metrics.csv для этого кадра (ключ связи).
- `sample_index` — абсолютный индекс отсчёта в accumulator (`start_index + k` из parse_cir).
- `I, Q` — из `parse_cir` (`samples[k]` = (i, q), int16).
- `amplitude` — `amps[k]` (sqrt(I²+Q²)).

Для каждого записанного кадра в полном режиме: взять текущий CIR (тот же, что уже
тянется для графика — `parse_cir`), выписать `count` строк (по числу отсчётов окна).
Плюс полезно продублировать в metrics.csv fp_index из CIR — НЕ обязательно; если
хочется, добавь колонку `CIR_fp_index` в конец metrics.csv (в отсчётах). На усмотрение
CC, но тогда задокументируй в заголовке/комментарии.

> Замечание по объёму (отразить в подсказке GUI): полный режим пишет ~33–61 строку
> CIR на кадр; при ~9 кадр/с это сотни строк CIR в секунду. Полный — для КОРОТКИХ
> прицельных записей (снять форму сигнала). Лёгкий — для длинных серий (статистика).

## 3. UI (добавить в существующее окно, не ломая текущее)

Блок «Запись» (например, под телеметрией или рядом с CIR):
- Переключатель режима записи: **Лёгкий** / **Полный** (radiobutton или combobox).
  По умолчанию — Лёгкий.
- Поле/кнопка выбора папки и префикса имени файла. При старте записи имена файлов
  формируются как `<префикс>_YYYYMMDD_HHMMSS_metrics.csv` (и `..._cir.csv` в полном),
  дата-время как у сохранения PNG (единый стиль). Если префикс пустой — использовать
  что-то по умолчанию (напр. `mks_rec`).
- Кнопка **Запись** (старт) / **Стоп записи**. Переключатель режима заблокировать во
  время активной записи (менять режим — только между записями).
- Индикация: «идёт запись» (напр. красная точка/текст), счётчик записанных кадров,
  и путь к файлу(ам). После стопа — показать итог («записано N кадров в <файл>»).

Запись работает во время детекции (rx активен, фоновый поток идёт). Если детекция не
запущена — кнопка «Запись» неактивна (нет данных). Останов детекции («Стоп») должен
также корректно останавливать запись (закрыть файлы).

## 4. Реализация (важные требования)

- **Источник данных — существующий фоновый поток.** НЕ добавлять второй опрос порта.
  Там, где поток уже получил распарсенные метрики (и CIR для графика) — если запись
  активна И `count` вырос с прошлого записанного кадра, дописать строку(и) в CSV.
- **Файлы открываются на старте записи, закрываются на стопе** (и при закрытии окна,
  и при остановке детекции). Использовать модуль `csv` (csv.writer) — он корректно
  экранирует и переносит строки. Флашить периодически или полагаться на закрытие;
  для надёжности при длинных записях — `flush()` время от времени (напр. раз в N кадров),
  чтобы данные не терялись при аварийном закрытии.
- **Потокобезопасность:** запись в файл делать в том же фоновом потоке, где берутся
  данные (не из главного/GUI-потока), чтобы не блокировать отрисовку и не гонять
  файловый объект между потоками. Счётчик записанных кадров и путь — передавать в GUI
  через ту же очередь/механизм, что и остальная телеметрия (обновление виджетов —
  только в главном потоке).
- **Ошибки записи** (нет прав на папку, диск и т.п.) — не ронять GUI: поймать,
  показать в статусной строке, остановить запись аккуратно.
- **Уникальность кадров:** хранить `last_written_count`; писать, только если текущий
  `count > last_written_count`; затем обновить. `frame_id` инкрементировать на каждую
  записанную строку metrics.
