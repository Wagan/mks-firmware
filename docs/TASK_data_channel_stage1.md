# Канал данных поверх UWB — §15.1 стадия (1): две радиостанции + фрагментация

**Дата:** 2026-07-22. **Основание:** RECON_data_channel_stage1.md (5746e12) + рамка
архитектора (вариант A: 6-байтовый заголовок с `src_id`). **Цель:** две независимые
радиостанции на РАЗНЫХ компьютерах (каждая своя плата/COM), текстовый чат с
фрагментацией >119 Б, индикатор связи для замера дальности в поле.

**Всё новое — на хосте (`tools\`). Прошивка/протокол НЕ менялись** (транспорт
`content=3` готов со стадии 0). Разбито на два коммита: логика (юнит-тестируемо) →
станция (приёмка на железе).

---

## Рамка архитектора (соблюдена)
- 6-байтовый заголовок: `src_id | msg_id | frag_idx | frag_cnt | msg_type | flags`,
  данные **119 Б**. `src_id` 1..255 (0 — резерв broadcast, не реализуется).
- Эхо-фильтр: приёмник отбрасывает кадр с `src_id == собственному` (кроме `--loopback`).
- Реассемблер по паре **(src_id, msg_id)** — msg_id двух станций пересекаются, разводит src_id.
- Индикатор «слышу партнёра» = слышали кадр с ЧУЖИМ src_id за 3 с; свои — не считаются.
- `msg_type`: 0=TEXT, 1=BEACON, 2=CALL. `flags=0` (иначе кадр нераспознан, не падаем).
- BEACON 1 с / порог 3 с; CALL ×3; пейсинга между фрагментами нет (TX_FRAME блокирует).

---

## Шаг 1 — логика `tools\mks_msg.py` (коммит: заголовок+фрагментация+реассемблер)
Чистый модуль (без COM), юнит-тестируемо:
- `pack_header/build_frame/parse_frame` — 6 Б заголовок; нераспознанный →
  `recognized=False` (короткий / `flags!=0` / неизвестный тип / `src=0` / `frag_idx>=cnt`).
- `fragment/fragment_text` — нарезка по 119 Б (пусто → 1 фрагмент); `decode_text`
  (UTF-8 `errors='replace'`).
- `is_own_echo(src, own, loopback)` — эхо-фильтр.
- `Reassembler` по `(src_id,msg_id)`: сборка; неполное — по новому msg_id того же src
  ИЛИ по таймауту (`missing k из n`, не склеивать молча); dedup завершённых (CALL ×3).

**Юниты (9 сценариев, все прошли):** парс TEXT/BEACON/CALL; нераспознанные; фрагментация
пусто/119/120/300; сборка байт-в-байт; потерянный фрагмент → incomplete; таймаут;
битый UTF-8 → replace; эхо-фильтр свой/чужой/loopback; **два источника с
пересекающимися msg_id не смешиваются**; CALL ×3 → одно completion.

## Шаг 2 — станция `tools\mks_station.py` (коммит: симметричная станция)
- Симметрична: один скрипт, отличие — `--port` и `--src-id`. ОДНОВРЕМЕННО принимает
  поток `content=3` (М2) и шлёт `TX_FRAME` (М1).
- **Один владелец COM-порта:** приём (`StreamReader`) и передача (`TX_FRAME` — командный
  кадр) в ОДНОМ engine-треде (pyserial не потокобезопасен); UI-тред кладёт исходящие
  кадры в очередь. После TX — `reader.buf.clear()` (ресинк).
- Эхо-фильтр по `src_id`; реассемблер; TEXT с фрагментацией; неполное с пометкой.
- BEACON каждую 1 с (**данные пусты** — RSSI позже, см. ниже); индикатор связи по
  чужому src за 3 с; beacon'ы в чат не печатаются.
- CALL ×3 → заметный алерт (звонок терминала `\a`) один раз (dedup).
- UI: `<текст>` отправить | `/call` | `/status` | `/quit`. Периодический статус раз в 5 с.

**Проверка (без железа):** py_compile + офлайн-smoke связки (исходящий текст 300→3
кадра и сборка; CALL ×3 dedup; эхо-фильтр свой/чужой; статус-строка; события
complete/incomplete/call) — **прошли**.

## Формат кадра канала данных (итог)
```
[ UWB-кадр, транспорт content=3 ]
src_id u8 | msg_id u8 | frag_idx u8 | frag_cnt u8 | msg_type u8 | flags u8 | data[<=119]
```
Заголовок — **прикладной слой ПОВЕРХ** `content=3` (для МКС это непрозрачные данные;
в `PROTOCOL_SPEC` не выносится — SPEC описывает протокол ПК↔МКС, а не хост-хост чат).

**Ограничение первого шага (по рамке):** BEACON несёт пустые данные — **RSSI пока
не показывается**. Причина: станция принимает `content=3` (только данные, без метрик);
RSSI живёт в `content=1`. Одновременно две content-моды одна плата не отдаёт. RSSI в
индикаторе — отдельный следующий шаг (напр. периодически дёргать `GET_SIGNAL_METRICS`
между кадрами, либо чередовать content). Индикатор СВЯЗЬ/НЕТ работает и без RSSI.

## Приёмка стадии (1) — на железе (ВАГАН)

### Шаг A — round-trip на ОДНОЙ плате (loopback, без второй машины)
```
python tools\mks_station.py COM3 --src-id 1 --loopback
```
- Ввести текст → он вернётся как `[1] <текст>` (М1→эфир→свой М2, эхо-фильтр выкл).
- Длинный текст (>119 симв.) → соберётся из нескольких фрагментов (проверить кириллицу,
  UTF-8 многобайтовый).
- `/call` → строка `*** ВЫЗОВ от станции 1 ***` + звонок терминала.
- Индикатор в loopback: партнёра нет (все кадры свои) → «СВЯЗЬ: — / ПОТЕРЯНА» — норма.

### Шаг B — две станции, две машины (полевой замер дальности)
Плата+ноутбук на каждом конце, свой COM, РАЗНЫЕ src_id, ОДИН Mode:
```
машина A:  python tools\mks_station.py COM3 --src-id 1 --name A
машина B:  python tools\mks_station.py COM4 --src-id 2 --name B
```
- Обмен текстом в обе стороны: у A печатается `[2] …`, у B `[1] …`. **Свои сообщения
  НЕ отражаются** (эхо-фильтр).
- Индикатор: пока станции слышат друг друга — `*** СВЯЗЬ ЕСТЬ ***` (и в `/status`).
- **Замер зоны:** отходить с ноутбуком B; на границе слышимости индикатор гаснет —
  `*** СВЯЗЬ ПОТЕРЯНА ***` (партнёрских beacon нет >3 с). Вернуться — связь восстановится.
- `/call` с A → у B заметный сигнал (даже если A вне «чата» — CALL ×3 повышает шанс
  без ARQ).
- Потеря фрагмента длинного текста (на границе связи) → `[src] <часть>  «НЕПОЛНОЕ:
  потерян фрагмент k из n»` (не склеивается молча).

**Отчёт Вагана:** round-trip loopback; двусторонний чат без само-эха; работает ли
индикатор как измеритель зоны (гаснет при уходе, загорается при возврате); слышен ли
CALL; как выглядит неполное сообщение.
