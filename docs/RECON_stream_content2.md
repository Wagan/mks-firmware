# RECON_stream_content2 — почему лёгкий поток (content=2) даёт 0 кадров

**Проект:** МКС / прошивка + хост (поток CIR-2)
**Автор задания:** Vagan Sarukhanov
**Исполнитель:** Claude Code (CC)
**Дата:** 2026-07-17
**Тип:** РАЗВЕДКА (recon), НЕ правка. Цель — локализовать причину, затем отдельным
шагом чинить. По правилу no-guessing: сначала найти, где именно ломается.

---

## 0. Контекст (факты из нагрузочного замера)

Матрица замеров (loopback M1→M2, киты выключены), поверх коммита 58a65e3:

- `content=1` (метрики+CIR) РАБОТАЕТ отлично: FPS растёт с частотой TX
  (50мс→19.5, 20мс→47, 10мс→90, 5мс→150), DROPPED(fw)=0, потери на хосте единичные.
- `content=2` (только метрики) — **0 кадров принято во всех прогонах** (периоды
  50/20/10/5 мс). При этом DROPPED=0, host_lost=0, crcErr=0, rx=0 — т.е. хост
  вообще не увидел ни одного валидного кадра (не «битые», а ПУСТО).

Команды прогонов content=2 (для повторения):
```
python mks_stream_probe.py COM3 --content 2 --txperiodic 50 --seconds 25
```
(аналогично 20/10/5)

Вывод из симптома: либо (А) прошивка при `stream_content==2` НЕ формирует/НЕ шлёт
кадр, либо (Б) хост-парсер не распознаёт кадр без CIR-блока и молча его отбрасывает
(не считая даже как crcErr). rx=0 + crcErr=0 больше похоже на «кадров физически нет
в потоке» ИЛИ «парсер отбрасывает до стадии CRC». Нужно точно определить, где.

## 1. Что проверить (recon, по коду — НЕ на железе)

CC работает по фактическому коду в репо (58a65e3). Проверить обе гипотезы:

### Гипотеза А — прошивка (protocol.c)

1. `HandleSET_STREAM_MODE`: при `mode==2` действительно ли ставится `stream_active=1`
   и `stream_content=2`? Нет ли условия, которое при content=2 оставляет
   `stream_active=0` или не шлёт?
2. `PROTOCOL_SendStreamFrame` (или где формируется кадр): при `stream_content==2`
   — кадр всё равно СОБИРАЕТСЯ и ОТПРАВЛЯЕТСЯ (CDC_Transmit_FS вызывается)? Нет ли
   ветки, которая при отсутствии CIR-блока делает `return` до отправки, или считает
   длину/CRC неверно (напр. LEN16=0, или PAYLOAD пустой → кадр не шлётся)?
3. Проверить формирование при content=2: PAYLOAD = только 30 байт метрик. LEN16
   тогда = SEQ(2)+DROPPED(2)+CONTENT(1)+30 = 35. Убедиться, что:
   - LEN16 считается от фактической длины (не «зашита» под content=1 с CIR);
   - CRC считается по правильной (короткой) длине;
   - буфер/указатели не уезжают (напр. не копируется cir_snap при content=2).
4. Зависит ли отправка при content=2 от `cir_snap.valid`? Если код шлёт кадр только
   когда снимок CIR готов — при content=2 CIR не нужен, но если условие осталось
   общим, а cir_snap по какой-то причине не valid — кадр не уйдёт. Проверить, нет ли
   такой связки.

### Гипотеза Б — хост (mks_stream_probe.py)

5. Парсер потокового кадра: как определяется длина кадра и наличие CIR? По CONTENT-
   байту. При CONTENT=2 парсер должен читать LEN16, взять ровно LEN16 байт тела,
   проверить CRC, разобрать 30 байт метрик и НЕ пытаться читать CIR.
   - Не «зашита» ли где-то ожидаемая длина под content=1 (с CIR)? Напр. парсер ждёт
     фиксированное число байт (с CIR) и при коротком кадре не находит конец/SMARK
     следующего → всё отбрасывает.
   - Правильно ли читается CONTENT (смещение после DROPPED)? Не читается ли CIR-
     заголовок безусловно (даже при CONTENT=2), уводя указатель?
6. Не блокирует ли что-то на старте именно при content=2: последовательность
   init→Mode3→rx_start→TX_PERIODIC→`SET_STREAM_MODE(2)`. Убедиться, что при
   `--content 2` реально шлётся `set_stream_mode(2)`, а не 1/0 (проверить проброс
   аргумента argparse → команда).
7. Юнит-smoke (он был в самопроверке CIR-2a-load): собрать эталонный кадр content=2
   в байтах и скормить парсеру probe — разберётся ли? Если smoke проходит, а железо
   даёт 0 — значит проблема в прошивке (кадр не шлётся) ИЛИ в приёме на хосте
   (тайминг/чтение), не в самом парсере.
