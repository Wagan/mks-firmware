## 1. Что делаем в Шаге 3

### 1.1. Порядок вкладок

### 1.2. Рендер водопада (вкладка «Водопад»)
Заменить заглушку реальным водопадом CIR — тепловая карта |CIR| во времени.

Дизайн (утверждён):
- **matplotlib imshow** (не строить самим) — 2D-массив амплитуд.
- **Ось X** — индекс отсчёта CIR (как на графике Монитора: start_index..).
- **Ось Y** — время/кадры. **Течёт ВНИЗ: свежий кадр — СВЕРХУ**, старые уходят вниз.
  (В imshow это означает: верхняя строка массива = самый свежий кадр; при добавлении
  нового — сдвиг вниз / вставка сверху. Проще: собирать массив так, чтобы row 0 =
  новейший, и `origin='upper'`.)
- **Цвет** — амплитуда |CIR| (sqrt(I²+Q²)), colormap на выбор CC (например 'viridis'
  или 'turbo' — читаемый; выбрать один, не давать переключатель сейчас).
- **Цветовая шкала — АВТОМАСШТАБ** по текущему содержимому буфера (vmin/vmax по
  фактическим данным окна). Colorbar показать. (Фиксированную шкалу — на будущее,
  не сейчас.)
- Данные — из буфера CIR (deque). Каждый кадр буфера = строка водопада (амплитуды по
  отсчётам окна). Если у кадров РАЗНАЯ длина окна/разный start_index (FP гуляет →
  окно сдвигается) — привести к общей сетке: см. §1.3.

### 1.3. Выравнивание кадров по оси X (важно — FP гуляет)
CIR-окно центрируется на first path, а fp_index от кадра к кадру гуляет (мы это
видели: 739–752). Значит start_index у разных кадров РАЗНЫЙ, и «строки» водопада
нельзя просто сложить в прямоугольник по позиции в массиве — они разъедутся.

Решение (выбрать одно, CC реализует наиболее простое надёжное):
- **(вариант А, рекомендуется) выравнивание по абсолютному sample_index.** Определить
  общий диапазон отсчётов по буферу [min(start_index) .. max(start_index+count-1)],
  построить матрицу этой ширины, каждый кадр положить в свои абсолютные позиции,
  пустые ячейки — NaN (imshow покажет их фоном). Тогда фронт CIR визуально «гуляет»
  по X от кадра к кадру — это ЧЕСТНО отражает дрожание FP, и для исследователя
  информативно.
- (вариант Б) выравнивание по FP: каждый кадр сдвинуть так, чтобы fp_index попал в
  один столбец (X = смещение относительно FP). Тогда фронт «стоит» на месте, видна
  форма хвоста. Тоже полезно, но сложнее и меняет смысл оси X.

CC: реализовать **вариант А** (абсолютный sample_index, NaN на пустых) — он проще,
честнее и не прячет дрожание FP. Вариант Б — возможное будущее переключение, НЕ сейчас.

### 1.4. Обновление водопада
- Обновлять с той же прорежённой частотой, что и график Монитора (~10–16/с; можно
  реже для водопада, напр. 5–10/с — перерисовка imshow тяжелее линии). Брать текущее
  содержимое буфера, строить матрицу, `imshow`. НЕ на каждый из 200 кадров.
- Обновлять, только когда вкладка «Водопад» АКТИВНА (видима) — если пользователь на
  Мониторе, водопад не перерисовывать (экономия). При переключении на вкладку —
  отрисовать текущий буфер.
- content=2 (нет CIR): водопад показывает заглушку «CIR отключён (content=2)» — как
  график CIR на Мониторе (уже сделано в Шаге 2, применить ту же логику к водопаду).
- Гашение: если поток молчит (нет кадров >1с) — водопад можно оставить как есть
  (последний буфер) ИЛИ пометить «нет сигнала». На выбор CC; проще — оставить буфер,
  он и так перестаёт пополняться. Не усложнять.
