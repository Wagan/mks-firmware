## 1. Приём payload — где оседает тело кадра

**Буфер и структура состояния (`protocol.c:185–197`):**
- `#define RX_FRAME_MAX 128` (`:185`) — потолок читаемого кадра.
- `static uint8_t rx_frame[RX_FRAME_MAX]` (`:197`) — буфер последнего принятого кадра.
- `static struct {...} rx_metrics` (`:189–195`): `valid u8`, `count u16`,
  **`frame_len u16` (`:192`)** — длина последнего кадра, `dwt_rxdiag_t diag`,
  `rxpacc_nosat u16`.

**Где кладётся тело — `PROTOCOL_PollRadio`, ветка RXFCG (`:1215–1229`):**
```c
frame_len = dwt_read32bitreg(RX_FINFO_ID) & RX_FINFO_RXFL_MASK_1023;  // :1219
rx_metrics.frame_len = frame_len;                                     // :1220
if (frame_len <= RX_FRAME_MAX) {                                      // :1221
    dwt_readrxdata(rx_frame, frame_len, 0);                           // :1222  тело сюда
}
```

**Ключевые факты:**
- `frame_len` — это **полная длина принятого кадра, ВКЛЮЧАЯ 2-байтный FCS** (802.15.4).
  Т.е. `rx_frame[0..frame_len-1]` = **payload + FCS(2)**. Чистый payload = `frame_len − 2`.
  (Симметрия к TX, где FCS дописывается аппаратно — см. §5.)
- **rx_frame НЕ обнуляется между кадрами.** Перетираются только первые `frame_len`
  байт. Если новый кадр короче предыдущего — «хвост» буфера содержит **старые байты**.
  ⇒ потребитель ОБЯЗАН читать ровно `frame_len` байт, не весь буфер.
- **Грабля переполнения (`:1221`):** при `frame_len > 128` чтение **пропускается**, а
  `rx_metrics.frame_len` всё равно записан = `frame_len`. Тогда `frame_len` врёт
  относительно содержимого `rx_frame` (в буфере — прошлый кадр). Для стандартного
  кадра (≤127) не срабатывает, но при отдаче наружу это надо учесть (проверять
  `frame_len ≤ RX_FRAME_MAX` перед использованием буфера).
- `valid=1` (`:1228`), `count++` (`:1229`) — уже после чтения тела.

**Наружу на хост `rx_frame` сейчас НЕ отдаётся:** нет ни команды-ответа, ни ветки
потока, которые бы его слали. Поток шлёт метрики(+CIR), не тело кадра (см. §2). Это и
есть недостающий фундамент стадии (0).

## 2. Точка отправки потока — `PROTOCOL_SendStreamFrame`

**Сигнатура:** `static void PROTOCOL_SendStreamFrame(void)` (`protocol.c:1120`).
Аргументов нет; активное DW-устройство (`DW_RX_LISTEN_DEV`) уже выбрано вызывающим.

**Где вызывается — `PROTOCOL_PollRadio`, RXFCG (`:1231–1237`):**
```c
PROTOCOL_CaptureCIR();               // :1231  снимок CIR ДО rxenable
if (stream_active) {
    PROTOCOL_SendStreamFrame();      // :1234  потоковый кадр ДО rxenable
}
dwt_rxenable(DWT_START_RX_IMMEDIATE);// :1237
```
Т.е. **один кадр на каждый принятый RXFCG**, без пейсинга (throttle — будущий CIR-2b).

**Сборка кадра (`:1120–1159`):**
| Шаг | Строки | Что |
|---|---|---|
| буфер | `:1122` | `static uint8_t frame[320]` («макс ~298 при полном снимке») |
| SMARK | `:1125` | `0xDE 0xCA` |
| LEN16 | `:1127` | резерв 2 байта (`len_field`, заполняется в конце) |
| SEQ | `:1129` | `stream_seq` u16 LE |
| DROPPED | `:1130` | `stream_dropped` u16 LE |
| CONTENT | `:1131` | `stream_content` u8 |
| метрики | `:1133` | `build_metrics_block(p)` → **30 байт ВСЕГДА** |
| CIR | `:1135–1142` | **только если `stream_content == 1`**: заголовок 6 Б (fp_index/start_index/count) + `memcpy(cir_snap.data, count*4)` |
| LEN16← | `:1144–1146` | `body_len = p − (len_field+2)`, вписывается в резерв |
| CRC8 | `:1149` | `PROTOCOL_CalculateCRC16len(len_field, p−len_field)` — по `[LEN16..конец PAYLOAD]`, **SMARK не входит** |
| отправка | `:1154–1157` | `CDC_Transmit_FS(frame,total)`: OK → `seq++`; иначе → `dropped++` |

**Ветвление по CONTENT:** сейчас ровно два пути — `content==1` добавляет окно CIR
(`:1135`), `content==2` не добавляет ничего (метрики уже вписаны на `:1133`). **PAYLOAD
вклинивается сразу после 30 байт метрик** (`p` после `build_metrics_block`). Ветки под
`content==3` нет — вставлять сюда же (после `:1142`).

**CRC:** отдельная функция `PROTOCOL_CalculateCRC16len` (`:104–113`) — тот же полином
`0x07`, но по длине `uint16` (тело потока может быть >255 из-за CIR). Командный CRC не
трогается.

## 3. CONTENT как перечисление — его нет

- **enum/define для content отсутствует.** `stream_content` — `static uint8_t`
  (`:233`), значения **1/2 — магические литералы** с комментариями (`:233`, `:947`,
  `:1131`). Дефолт `stream_content = 1` в `PROTOCOL_Init` (`:264`).
- **Валидация `SET_STREAM_MODE` (`HandleSET_STREAM_MODE:939–941`):**
  ```c
  if (params_len < 1) return STATUS_INVALID_PARAM;   // :939  нет байта mode
  uint8_t mode = params[0];
  if (mode > 2) return STATUS_INVALID_PARAM;          // :941  отвергает mode>2
  ```
  Т.е. `INVALID_PARAM` даётся на **любой mode ≥ 3** (и на пустые params). Принимаются
  0/1/2. `stream_content = mode` (`:947`) — присваивание **уже обобщённое**.
- **Стоимость добавления `content=3` — низкая, 3 точки, без структурных изменений:**
  1. `:941` — поднять порог: `mode > 2` → `mode > 3`.
  2. `PROTOCOL_SendStreamFrame` после `:1142` — добавить `if (stream_content == 3) {…}`
     (вписать payload данных, напр. `frame_len−2` байт из `rx_frame`).
  3. `stream_content = mode` (`:947`) менять не надо — уже принимает любой mode.
  Плюс, возможно, увеличить `frame[320]` под размер payload (для тела ≤128 — не нужно,
  запас есть; см. §4).

## 4. Бюджет кадра

**Максимум при `content=1`** (метрики 30 + окно CIR до 63 отсч. × 4 = 252):
```
SMARK 2 + LEN16 2 + SEQ 2 + DROPPED 2 + CONTENT 1 + метрики 30
      + CIR_hdr 6 + CIR_data 252 + CRC 1  =  298 байт
```
(совпадает с комментарием `:1122`; `CIR_HALF_MAX=31 → 63 отсч.`, `:217–219`.)
- `frame[320]` → **запас ~22 байта** над текущим максимумом.
- Тело под LEN16 (content=1 max) = `2+2+1+30+6+252 = 293` → **LEN16 (u16) свободно
  вмещает** (потолок 65535).
- **Фиксированный оверхед кадра** = `SMARK2+LEN16 2+SEQ2+DROPPED2+CONTENT1+CRC1 = 10 байт`.

**Реальный потолок под payload данных — НЕ LEN16 и НЕ USB, а статический `frame[320]`:**
- LEN16 — u16 (до 65535), далеко.
- USB: `CDC_Transmit_FS` (`usbd_cdc_if.c:286–298`) кладёт `Len` (uint16) через
  `USBD_CDC_SetTxBuffer(..., Buf, Len)` → указывает **напрямую на переданный `frame`**,
  стек сам дробит на 64-байтные пакеты. **`UserTxBufferFS[APP_TX_DATA_SIZE=2048]`
  (`usbd_cdc_if.h:52`, `:94`) в этом пути НЕ используется** — так что «2048» здесь не
  ограничение. Практический потолок одной посылки — размер буфера вызывающего.
- ⇒ **упрёмся в `frame[320]` раньше всего.** Для content=3, несущего тело принятого
  кадра (≤ `RX_FRAME_MAX=128`): `10 + 128 = 138 ≤ 320` — **влезает с запасом, буфер
  расширять не нужно.** Для payload крупнее ~310 — увеличить `frame[]`.
- **Обратное давление:** `CDC_Transmit_FS` при `TxState!=0` возвращает `USBD_BUSY`
  (`:291–293`) → кадр **дропается** (`dropped++`, `:1157`), не буферизуется и не блокирует.
  На высоком темпе это штатные потери (SEQ непрерывен, DROPPED растёт).

## 5. TX для ответной ветки (симметрия «хост→радио» vs «радио→хост»)

**`HandleTX_FRAME` (`:982–1014`):**
- Параметры (wire): `length u16 LE` (`:989`) + `payload[length]` (`&params[2]`).
- Валидация: `params_len<2` → `INVALID_PARAM` (`:988`); `params_len<2+length` →
  `INVALID_PARAM` (`:990`); `length > TX_FRAME_MAX(125)` → `BUFFER_OVERFLOW` (`:991`).
- Передача: `dwt_writetxdata(length+2, &params[2], 0)` (`:999`) — пишет **чистые
  `length` байт payload**, `+2` резерв под **авто-FCS** (дописывает DW1000);
  `dwt_writetxfctrl(length+2,0,0)` (`:1001`); `dwt_starttx` (`:1003`); busy-wait TXFRS
  с guard (`:1006–1011`).

**Симметрия путей:**
| | хост→радио (TX_FRAME) | радио→хост (стадия 0) |
|---|---|---|
| носитель | length-prefixed байтовый буфер | тот же по форме (LEN16+тело в потоке) |
| payload | **чистый**, ≤125 (FCS дописывает железо) | `rx_frame`, `frame_len` байт = **payload + FCS(2)** |
| чтобы получить payload | — | **отрезать 2 байта FCS** (`frame_len − 2`) |
| макс | 125 | payload ≤ `frame_len−2` (буфер 128; станд. кадр → ≤125) |

⇒ Путь **симметричен по форме** (длина + байты), но **асимметричен по FCS**: TX
отдаёт чистый payload, а в `rx_frame` FCS присутствует. Для стадии (0) — решить
однозначно: слать `frame_len−2` (payload, симметрично TX) либо `frame_len` (сырое с
FCS, для учебной наглядности CRC-контроля). Максимальные размеры сходятся (125 ↔ ≤125).

## 6. Расхождения с PROTOCOL_SPEC v1.5 §8 (прав код)

- **Материальных расхождений НЕТ.** Формат потока, покрытие CRC (`[LEN16..PAYLOAD]`,
  SMARK вне), семантика SEQ/DROPPED, метрики 30 Б (общий `build_metrics_block`) —
  совпадают с §8.
- **Нюанс окна CIR (не противоречие, но отметить):** поток снимает окно с
  `CIR_HALF_MAX=31` → **до 63 отсчётов** (`:217–219`), а команда `GET_CIR` ограничена
  `half≤30` → **до 61 отсчёта** (`CIR_RESP_HALF_MAX=30`, `:876`). SPEC это уже
  отражает (в §8 SET_STREAM_MODE — «полуширина захвата ≈ 31 → до 63», в GET_CIR — «макс
  30»), т.е. два окна **разной ширины** — сознательно.
- **Пробел (ожидаемый):** SPEC не описывает никакого способа отдать `rx_frame` наружу —
  и в коде его нет. Это ровно цель стадии (0); при реализации SPEC §8 дополнить
  описанием content-режима «данные».

## 7. Вывод для стадии (0)

Фундамент отсутствует ровно в одном месте: **тело принятого кадра (`rx_frame`,
`frame_len` байт) читается, но не выводится наружу.** Инфраструктура готова:
- потоковый механизм (`PROTOCOL_SendStreamFrame`) — есть, ветвление по `content` — есть,
  добавление `content=3` дёшево (3 точки, §3);
- бюджет кадра — с запасом (`frame[320]` вмещает `10 + 128`, §4);
- USB-путь — тянет (стек дробит сам, потолок = `frame[]`, не 64/2048);
- обратный путь TX — симметричен по форме, с поправкой на FCS (§5).

Открытые решения на согласование перед кодом: (а) слать `frame_len−2` или сырое с FCS;
(б) нужен ли отдельный заголовок длины payload внутри PAYLOAD (при content=3 тело
переменной длины — LEN16 её и так задаёт, но явное поле упростит парсер хоста);
(в) отдавать данные ТОЛЬКО при content=3 или разрешить «данные + метрики».
