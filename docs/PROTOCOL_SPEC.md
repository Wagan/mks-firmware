**Версия:** 1.4 (первая редакция, приведённая к реализации)
**Дата:** 2026-07-16
**Статус:** живой документ. Обновляется вслед за кодом.
**Проверено на железе (2026-07-16):** команды `PING` / `INIT` / `GET_STATUS`

PING→PONG, INIT→OK (оба модуля DWM1000 инициализированы), GET_STATUS→7 байт.

**Передающий тракт проверен на железе (2026-07-17):** `TX_FRAME` / `TX_STOP`

RXPACC 264, FP_AMPL ~16000, RX_LEVEL −78.8 dBm, diff 2.2 дБ → LOS). Замкнута
доверенная пара свой TX (M1) ↔ свой RX (M2), PHY согласован через `mode3`.

## 1. Физический интерфейс

- Транспорт: **USB CDC (Virtual COM Port)** поверх USB FS.
- Параметры порта: **115200 8N1** (базовая скорость; CDC допускает выше).
- На ПК: любой инструмент, открывающий COM-порт (Python `pyserial`, MATLAB
  `serialport()` и т.п.).
- На стороне МКС порт определяется ОС как «STM32 Virtual COM Port».

---

## 2. Структура пакета команды (ПК → МКС)

```
+--------+-------+--------+------------------+-------+
| SYNC   | LEN   | CMD_ID | PARAMS           | CRC   |
| 2 байта| 1 байт| 1 байт | 0..255 байт      | 1 байт|
+--------+-------+--------+------------------+-------+
```

| Поле | Размер | Описание |
|---|---|---|
| SYNC | 2 | Маркер начала кадра. Байты на проводе: **`0xAA` затем `0x55`**. |
| LEN | 1 | Длина `CMD_ID + PARAMS` (т.е. 1 + число байт параметров). |
| CMD_ID | 1 | Код команды (см. §5). |
| PARAMS | 0..255 | Параметры команды (little-endian, см. §6). |
| CRC | 1 | CRC8 по **LEN + CMD_ID + PARAMS** (см. §4). |

> **Порядок SYNC (важно).** На проводе первым идёт `0xAA`, затем `0x55`
> (величина 0xAA55 записана старшим байтом вперёд именно как последовательность
> `AA 55`). Приёмник ищет пару `AA 55`.

---

## 3. Структура ответа (МКС → ПК)

```
+--------+-------+--------+------------------+-------+
| SYNC   | LEN   | STATUS | DATA             | CRC   |
| 2 байта| 1 байт| 1 байт | 0..N байт        | 1 байт|
+--------+-------+--------+------------------+-------+
```

| Поле | Размер | Описание |
|---|---|---|
| SYNC | 2 | `0xAA` затем `0x55`. |
| LEN | 1 | Длина `STATUS + DATA`. |
| STATUS | 1 | Код результата (§7). `0x00` = успех. |
| DATA | 0..N | Данные ответа (зависит от команды, §8). |
| CRC | 1 | CRC8 по **LEN + STATUS + DATA**. |

---

## 4. CRC8

- Алгоритм: **CRC-8**, полином **`0x07`**, начальное значение **`0x00`**, без
  входной/выходной рефлексии, без финального XOR (xorout = 0x00).
- **Область покрытия: `LEN + CMD_ID + PARAMS`** (для команды) / `LEN + STATUS +
  DATA` (для ответа). **SYNC в CRC НЕ входит.**

> **Расхождение с исходным PDF.** Старый «Бинарный протокол v1.3» в тексте писал
> «CRC по всем предыдущим байтам» и приводил MATLAB-пример, где CRC считался
> вместе с SYNC. Это признано ошибочным. Принято решение: **CRC без SYNC.** В
> прошивке область покрытия вынесена в макрос (`PROTOCOL_CRC_COVERS_SYNC=0`) на
> случай, если понадобится совместимость.

Референс-реализация (Python) — см. §9.

## 6. Форматы параметров команд

Типы: `uint8` (1Б), `uint16` (2Б, LE), `uint32` (4Б, LE), `int16` (2Б, LE).
Все многобайтовые значения — **little-endian**.

| CMD_ID | Команда | Параметры |
|---|---|---|
| 0x00 | PING | нет |
| 0x01 | INIT | нет |
| 0x02 | GET_STATUS | нет |
| 0x03 | RESET_RADIO | нет |
| 0x10 | SET_PHY_CONFIG | channel u8, data_rate u8, preamble_length u16, preamble_code u8, PRF u8, PAC_size u8 |
| 0x11 | SET_TX_POWER | power_level u8 |
| 0x20 | TX_FRAME | length u16, payload[length] |
| 0x21 | TX_PERIODIC | period_ms u16, length u16, payload[length] |
| 0x22 | TX_STOP | нет |
| 0x30 | RX_START | нет |
| 0x31 | RX_STOP | нет |
| 0x40 | GET_SIGNAL_METRICS | нет |
| 0x41 | GET_CIR | half u8 (полуширина окна вокруг first path; 0 = дефолт 16, макс. 30; центрирование по FP делает прошивка) |
| 0x42 | SET_STREAM_MODE | mode u8 (0=выкл, 1=вкл метрики+CIR, 2=вкл только метрики) |
| 0x50 | START_EXPERIMENT | нет |
| 0x51 | STOP_EXPERIMENT | нет |
| 0x60 | TX_SWEEP | channel_start u8, channel_end u8, power_start u8, power_end u8, preamble_length u16 |
| 0x61 | DETECTOR_TEST | number_of_packets u16, tx_power_start u8, tx_power_end u8 |

> Значения `preamble_length`, `PRF`, `PAC_size` передаются как «сырые» числа
> (симв./МГц/симв.); прошивка транслирует их в enum DecaDriver с валидацией.
> Невалидное значение → `INVALID_PARAM`. Таблицы трансляции:
>
> - **channel** (u8): допустимые `{1,2,3,4,5,7}`, пропускается как есть.
> - **data_rate** (u8): **КОД** `{0:110 kbps, 1:850 kbps, 2:6.8 Mbps}` →
>   `DWT_BR_110K/850K/6M8`. Именно код, а не значение в кбит/с: 850 и 6800 не
>   помещаются в u8.
> - **preamble_length** (u16): `64/128/256/512/1024/1536/2048/4096` → `DWT_PLEN_*`.
> - **preamble_code** (u8): `1..24` (широкая валидация; строгая PRF-зависимая —
>   позже). Задаёт и `txCode`, и `rxCode`.
> - **PRF** (u8): число МГц `16/64` → `DWT_PRF_16M/64M`.
> - **PAC_size** (u8): число символов `8/16/32/64` → `DWT_PAC8/16/32/64`.
>
> **nsSFD** (нестандартный SFD) протоколом не передаётся: прошивка ставит его по
> правилу `data_rate == 110 kbps → nsSFD = 1` (иначе 0), по конвенции EVK/DecaWave.
> `sfdTO` на первом этапе = 0 → драйверный дефолт `DWT_SFDTOC_DEF`. `phrMode` =
> `DWT_PHRMODE_STD`.

## 7. Коды STATUS (ответ МКС)

| STATUS | Имя | Описание |
|---|---|---|
| 0x00 | OK | Команда выполнена успешно |
| 0x01 | UNKNOWN_CMD | CMD_ID отсутствует в таблице / обработчик не зарегистрирован |
| 0x02 | INVALID_PARAM | Параметры вне допустимого диапазона |
| 0x03 | RADIO_BUSY | Радиомодуль занят (например, TX уже активен) |
| 0x04 | RADIO_ERROR | DW1000 вернул ошибку |
| 0x05 | BUFFER_OVERFLOW | payload больше допустимого / переполнение буфера |
| 0x06 | TIMEOUT | Таймаут операции (например, ожидание RX) |
| 0x07 | INTERNAL_ERROR | Внутренняя ошибка прошивки (в т.ч. ошибка CRC запроса) |

> При несовпадении CRC входящего пакета прошивка отвечает кодом ошибки
> (`INTERNAL_ERROR`), а не игнорирует пакет молча.

## 8. Формат данных ответа (DATA) — реализованные команды

### PING (0x00) — ✅
DATA: **1 байт `0x00`** (PONG). LEN = 2 (STATUS + 1 байт DATA).

> Приведено к «Бинарный протокол §11»: для PING формат ответа — `OK uint8`
> (один байт данных). API допускал и пустой DATA («или просто код успеха»), но
> берём строгий вариант протокола — 1 байт.

### INIT (0x01) — ✅
**Обнаруживает живые модули по DEV_ID.** Для каждого места: reset → медленный SPI →
проверка `dwt_readdevid() == DWT_DEVICE_ID (0xDECA0130)`; если модуль отвечает —
`dwt_initialise` (+ LDE-микрокод) и пометка живым. Отсутствующий/немой модуль
**пропускается** (не роняет общий статус).

- STATUS = `OK`, если жив **≥1** модуль; DATA (2 байта): `live_count u8`,
  `live_mask u8` (бит i = модуль i живой; бит0=M1, бит1=M2).
- STATUS = `RADIO_ERROR`, если живых нет.

Логика обобщена под `DW_DEVICE_COUNT` (заработает и на будущем 5-канальном макете).
Команды затем работают только с живыми ролями (проверка `.initialized`): на плате
с одним M2 приём/поток идут, а `TX_*` честно возвращают `RADIO_ERROR`.

### GET_STATUS (0x02) — ✅
Первые **7 байт — прежний формат** (байт-в-байт, из кэша по M1); далее аддитивное
расширение (живость модулей). Итог для 2 модулей — **17 байт**.

| Смещение | Поле | Тип | Прим. |
|---|---|---|---|
| 0 | TX_state | u8 | пока 0 (трекинга TX/RX ещё нет) |
| 1 | RX_state | u8 | пока 0 |
| 2 | channel | u8 | из кэша (0 до SET_PHY_CONFIG), по M1 |
| 3 | data_rate | u8 | из кэша |
| 4..5 | preamble_length | u16 LE | из кэша |
| 6 | PRF | u8 | из кэша |
| 7 | live_count | u8 | число живых модулей |
| 8 | live_mask | u8 | бит i = модуль i живой |
| 9..12 | dev_id[0] (M1) | u32 LE | 0 если не живой |
| 13..16 | dev_id[1] (M2) | u32 LE | 0 если не живой |

Старый хост читает первые 7 байт и не ломается. Расширение — `2 + DW_DEVICE_COUNT*4`
байт (для 5 модулей — по одному `dev_id` на модуль).

### TX_FRAME (0x20) / TX_STOP (0x22) — ✅ 🔬
`TX_FRAME` params (§6): `length u16 (LE)`, `payload[length]`. **payload — БЕЗ FCS**:
DW1000 дописывает 2-байтный FCS сам, поэтому в драйвер передаётся `length + 2`
(`dwt_writetxdata`/`dwt_writetxfctrl`; сам driver копирует `len−2` = payload).
Передаёт с модуля `DW_TX_SOURCE_DEV` (board_config; по умолчанию M1), требует
предварительного INIT. Ждёт `TXFRS` (кадр ушёл) с guard-счётчиком. DATA нет:
`STATUS_OK` при пойманном `TXFRS`, иначе `RADIO_ERROR` (ошибка writetxdata/starttx)
или `TIMEOUT` (TXFRS не дождались). Лимит payload — `TX_FRAME_MAX = 125` (127 −
2 FCS), иначе `BUFFER_OVERFLOW`. `TX_STOP` — `dwt_forcetrxoff` на модуле-источнике.
Проверено на железе (2026-07-17): loopback M1→эфир→M2 при выключенном EVK, count=1.

### RX_START (0x30) / RX_STOP (0x31) — ✅ 🔬
Параметров нет. DATA нет. `RX_START` включает непрерывный приём на модуле
`DW_RX_LISTEN_DEV` (board_config; по умолчанию M2), требует предварительного INIT
(иначе `RADIO_ERROR`). Приём и разбор кадров — в main loop (`PROTOCOL_PollRadio`,
опрос SYS_STATUS; EXTI добавим позже). `RX_STOP` — `dwt_forcetrxoff`.
Проверено на железе (2026-07-16): M2 расслышал EVK1000 Mode 3, свап на M1 не понадобился.

### GET_SIGNAL_METRICS (0x40) — ✅ 🔬 ⚠️ ИНТЕРИМ-формат
Пока кадра не принято → `TIMEOUT`. Иначе DATA (**18 байт, u16 LE**) — сырые поля
`dwt_rxdiag_t` последнего кадра:

| Смещение | Поле | Источник (dwt_rxdiag_t) |
|---|---|---|
| 0..1 | count (число принятых кадров) | счётчик прошивки (отладка) |
| 2..3 | CIR_PWR | `maxGrowthCIR` |
| 4..5 | RXPACC | `rxPreamCount` |
| 6..7 | STD_NOISE | `stdNoise` |
| 8..9 | FP_AMPL1 | `firstPathAmp1` |
| 10..11 | FP_AMPL2 | `firstPathAmp2` |
| 12..13 | FP_AMPL3 | `firstPathAmp3` |
| 14..15 | FP_INDEX | `firstPath` |
| 16..17 | MAX_NOISE | `maxNoise` |

> ⚠️ **ИНТЕРИМ.** Это сырые поля для проверки факта приёма и как входы для
> формул. Итоговый формат — RSSI int16 (dBm×100), SNR int16 (dB×100), RXPACC u16,
> FP_INDEX u16 — по формулам **DW1000 UM §4.7** (заменит интерим).
>
> **Находка для перехода к RSSI (по замерам EVK Mode 3, преамбула 1024):** RXPACC
> замерен ≈ **261–280** — счётчик насытился рано, т.е. `RXPACC ≠ RXPACC_NOSAT`,
> значит по UM §4.7 SFD-коррекция N **не требуется**. Но строго это подтвердить
> нельзя: `RXPACC_NOSAT` в интерим-формате отсутствует. Для RSSI (UM §4.7.2:
> `RX_LEVEL = 10·log10(C·2^17 / N²) − A`, где `A = 113.77 @PRF16 / 121.74 @PRF64`,
> `C = CIR_PWR`, `N = RXPACC`) **нужно добавить чтение `RXPACC_NOSAT`** (из
> `DRX_CONF`/`RXPACC_NOSAT`), чтобы корректно брать N.

> ✅ **Хостовая приближённая оценка мощности реализована и проверена (2026-07-17).**
> В `tools/mks_protocol.py` (`estimate_power`) и выводе команды `metrics`
> `tools/mks_console.py` по формулам UM §4.7 считаются `RX_LEVEL` (RSSI) и
> `FP_POWER`, `diff = RX_LEVEL − FP_POWER` и классификатор LOS/gray/NLOS (пороги
> UM §4.7.1: `<6 дБ` LOS, `>10 дБ` NLOS). Константы `A = 113.77` (PRF 16М) /
> `121.74` (PRF 64М). Проверено на живом EVK Mode 3: `RX_LEVEL` стабилен в
> −80…−82 dBm, `diff` реагирует на смену канала (1.3…8.9 дБ), классификатор
> работает.
> ⚠️ **Расчёт ПРИБЛИЖЁННЫЙ:** `N` (RXPACC) берётся **без SFD-коррекции**, т.к.
> `RXPACC_NOSAT` в интерим-формате отсутствует. Строгий RSSI/SNR и **перенос
> формул в прошивку** — отдельный будущий шаг, после добавления чтения
> `RXPACC_NOSAT`. Пока формулы живут только на хосте; в код прошивки НЕ внесены.

> **Наблюдение для будущего строгого SNR (loopback M1→M2, 2026-07-17):** внутренний
> loopback заметно **шумнее** эфирного канала с EVK — `STD_NOISE ~164` (против
> ~56–84 на EVK), `MAX_NOISE ~4139` (против ~1100–2000). Вероятная причина —
> наводки между модулями на одной плате + кабели RG-58. Тем не менее headroom
> FP/noise ≈ 4×, приём уверенный. Учесть при строгом SNR (шумовой пол зависит от
> источника сигнала: свой TX vs внешний эфир).

### GET_CIR (0x41) — ✅ 🔬 (шаг CIR-1: усечённое окно вокруг first path)
Параметр (wire): `half u8` — полуширина окна (0 → дефолт 16; клампится к макс. 30).
Центр окна — **first path** (индекс `firstPath>>6`), выбирает **прошивка**. Снимок
окна CIR берётся из accumulator в ветке приёма (RXFCG) ДО перевзвода RX; `GET_CIR`
отдаёт из снимка. Если кадра/снимка ещё не было → `TIMEOUT`.

DATA — заголовок 6 байт + тело:

| Смещение | Поле | Тип |
|---|---|---|
| 0..1 | fp_index | u16 LE (индекс first path в accumulator) |
| 2..3 | start_index | u16 LE (индекс первого отсчёта в ответе) |
| 4..5 | count | u16 LE (число отсчётов далее) |
| 6..  | I/Q пары | int16 LE × 2 × count (I1,Q1,I2,Q2,…) |

Итого `6 + count*4` байт. Лимит: `6 + count*4 ≤ 255` (LEN — 1 байт) → `count ≤ 62`,
поэтому `half ≤ 30` (61 отсчёт, 250 байт).

> Это **усечённое** окно (диагностика формы first path). Полный CIR (1016 отсчётов)
> одним ответом не влезает в 255-байтный кадр; полный CIR / потоковый режим /
> водопад и произвольный срез (offset/length) — будущие шаги CIR-2/3.

> **Проверено на железе (2026-07-17):** `init → mode3 → rxstart → приём → cir` дало
> корректный результат. `fp_index` в домене отсчётов (746–749); резкий фронт CIR у
> first path (шумовой пол ~сотни до FP, рост до ~16000–17000 за 3 отсчёта), хвост
> многолучёвки за пиком. Маркер FP стоит на **переднем крае** нарастающего фронта
> (LDE leading-edge — на 2–3 отсчёта раньше максимума пика; корректно: детектор ищет
> первый путь, а не максимум). Снимок берётся в ветке RXFCG **до `dwt_rxenable`** —
> грабля затирания accumulator подтверждённо снята (иначе были бы нули/мусор).

### SET_STREAM_MODE (0x42) — ✅ + Потоковый режим (CIR-2a)
Параметр: `mode u8` (0=выкл/командный; 1=вкл метрики+CIR; 2=вкл только метрики).
Ответ на саму команду — DATA нет (`OK`/`INVALID_PARAM`). При `mode≥1` плата **после
каждого принятого UWB-кадра** сама шлёт по тому же USB CDC **потоковый кадр** (без
запроса хоста); при `mode=0` — обычный командный режим (дефолт после INIT).

**Потоковый кадр — СВОЙ формат, отдельный от командного** (командный
`SYNC 0xAA55 | LEN | CMD | PARAMS | CRC` не затронут):

```
SMARK(0xDE 0xCA) | LEN16(u16 LE) | SEQ(u16) | DROPPED(u16) | CONTENT(u8) | PAYLOAD | CRC8
```
- `SMARK` = `0xDE 0xCA` — маркер начала (отличен от командного SYNC; для
  ре-синхронизации хоста).
- `LEN16` — число байт **после LEN16 и до CRC** (= SEQ+DROPPED+CONTENT+PAYLOAD), u16 LE.
- `SEQ` u16 — номер **отправленного** кадра (инкремент только на реально ушедший;
  дырки в SEQ на хосте = потери на пути/у хоста). Оборачивается (16 бит).
- `DROPPED` u16 — счётчик кадров, **дропнутых прошивкой при CDC BUSY** (нарастающий,
  оборачивается). Растёт при непрерывном SEQ = прошивка не успела отдать в USB.
- `CONTENT` u8 — что в PAYLOAD: **1** = метрики+окно CIR; **2** = только метрики
  (лёгкий поток для нагрузочного замера / когда CIR не нужен).
- `PAYLOAD` = **метрики 30 байт** (тот же формат, что `GET_SIGNAL_METRICS`); при
  `CONTENT=1` дополнительно **окно CIR** (заголовок 6 байт `fp_index/start_index/
  count` + `count` пар I/Q int16 LE — как DATA у `GET_CIR`; весь снимок `cir_snap`,
  фикс. полуширина захвата ≈ 31 → до 63 отсчётов).
- `CRC8` — тот же полином `0x07` по `[LEN16 .. конец PAYLOAD)` (SMARK не входит).

Политика при **CDC BUSY**: кадр **дропается** (не повторяется), `DROPPED++`, `SEQ`
не трогается. Приёмник: `tools/mks_stream_probe.py` (замер FPS/потерь; лёгкий режим
`--content 2` + `--txperiodic` loopback для нащупывания потолка USB). Полный CIR /
bulk / интеграция в GUI — будущие подшаги.

## 9. Референс CRC8 (Python)

```python
def crc8(data: bytes) -> int:
    # CRC-8, poly 0x07, init 0x00, no reflection, xorout 0x00
    crc = 0
    for b in data:
        crc ^= b
        for _ in range(8):
            crc = ((crc << 1) ^ 0x07) & 0xFF if (crc & 0x80) else (crc << 1) & 0xFF
    return crc
```

Область: для команды `crc8(bytes([LEN, CMD_ID, *PARAMS]))` — **без SYNC**.

## 10. Примеры кадров

**PING** (запрос):
```
AA 55        SYNC
01           LEN  (CMD_ID + PARAMS = 1)
00           CMD_ID = PING
<crc>        CRC8 по [01, 00]
```
Ответ:
```
AA 55        SYNC
02           LEN  (STATUS + DATA = 2)
00           STATUS = OK
00           DATA   = PONG
<crc>        CRC8 по [02, 00, 00]
```

**GET_STATUS** (запрос): `AA 55 01 02 <crc>` (CRC по `[01,02]`).
