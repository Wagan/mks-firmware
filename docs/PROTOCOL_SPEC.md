# PROTOCOL_SPEC — протокол обмена ПК ↔ МКС

**Версия:** 1.4 (первая редакция, приведённая к реализации)
**Дата:** 2026-07-16
**Статус:** живой документ. Обновляется вслед за кодом.
**Проверено на железе (2026-07-16):** команды `PING` / `INIT` / `GET_STATUS`
проверены end-to-end на плате МКС по USB CDC (COM3) через `tools/mks_*_test.py`:
PING→PONG, INIT→OK (оба модуля DWM1000 инициализированы), GET_STATUS→7 байт.
**Приёмный тракт проверен на железе (2026-07-16):** `RX_START` / `GET_SIGNAL_METRICS`
проверены против эталонного передатчика **EVK1000 (Mode 3, 0.7 м LOS)** через
`tools/mks_console.py`. Последовательность INIT→SET_PHY_CONFIG(Mode3)→RX_START→
metrics дала растущий `count` (48→81→138) и физичные метрики (CIR_PWR ~9000,
RXPACC ~270, FP_AMPL1..3 ~16000–18000). Слушающий модуль — M2 (по умолчанию).

> **Источник истины — реализация.** Этот документ описывает то, что прошивка МКС
> реально делает. Исходные PDF («Бинарный протокол v1.3», «API v1.3») были
> стартовым ориентиром и перенесены в `docs/archive/` как исторические — они НЕ

> любого документа — прав код; документ приводится к коду.

## 1. Физический интерфейс

- Транспорт: **USB CDC (Virtual COM Port)** поверх USB FS.
- Параметры порта: **115200 8N1** (базовая скорость; CDC допускает выше).
- На ПК: любой инструмент, открывающий COM-порт (Python `pyserial`, MATLAB
  `serialport()` и т.п.).
- На стороне МКС порт определяется ОС как «STM32 Virtual COM Port».

---

## 2. Структура пакета команды (ПК → МКС)

```
+--------+-------+--------+------------------+-------+
| SYNC   | LEN   | CMD_ID | PARAMS           | CRC   |
| 2 байта| 1 байт| 1 байт | 0..255 байт      | 1 байт|
+--------+-------+--------+------------------+-------+
```

| Поле | Размер | Описание |
|---|---|---|
| SYNC | 2 | Маркер начала кадра. Байты на проводе: **`0xAA` затем `0x55`**. |
| LEN | 1 | Длина `CMD_ID + PARAMS` (т.е. 1 + число байт параметров). |
| CMD_ID | 1 | Код команды (см. §5). |
| PARAMS | 0..255 | Параметры команды (little-endian, см. §6). |
| CRC | 1 | CRC8 по **LEN + CMD_ID + PARAMS** (см. §4). |

> **Порядок SYNC (важно).** На проводе первым идёт `0xAA`, затем `0x55`
> (величина 0xAA55 записана старшим байтом вперёд именно как последовательность
> `AA 55`). Приёмник ищет пару `AA 55`.

---

## 3. Структура ответа (МКС → ПК)

```
+--------+-------+--------+------------------+-------+
| SYNC   | LEN   | STATUS | DATA             | CRC   |
| 2 байта| 1 байт| 1 байт | 0..N байт        | 1 байт|
+--------+-------+--------+------------------+-------+
```

| Поле | Размер | Описание |
|---|---|---|
| SYNC | 2 | `0xAA` затем `0x55`. |
| LEN | 1 | Длина `STATUS + DATA`. |
| STATUS | 1 | Код результата (§7). `0x00` = успех. |
| DATA | 0..N | Данные ответа (зависит от команды, §8). |
| CRC | 1 | CRC8 по **LEN + STATUS + DATA**. |

---

## 4. CRC8

- Алгоритм: **CRC-8**, полином **`0x07`**, начальное значение **`0x00`**, без
  входной/выходной рефлексии, без финального XOR (xorout = 0x00).
- **Область покрытия: `LEN + CMD_ID + PARAMS`** (для команды) / `LEN + STATUS +
  DATA` (для ответа). **SYNC в CRC НЕ входит.**

> **Расхождение с исходным PDF.** Старый «Бинарный протокол v1.3» в тексте писал
> «CRC по всем предыдущим байтам» и приводил MATLAB-пример, где CRC считался
> вместе с SYNC. Это признано ошибочным. Принято решение: **CRC без SYNC.** В
> прошивке область покрытия вынесена в макрос (`PROTOCOL_CRC_COVERS_SYNC=0`) на
> случай, если понадобится совместимость.

Референс-реализация (Python) — см. §9.

## 6. Форматы параметров команд

Типы: `uint8` (1Б), `uint16` (2Б, LE), `uint32` (4Б, LE), `int16` (2Б, LE).
Все многобайтовые значения — **little-endian**.

| CMD_ID | Команда | Параметры |
|---|---|---|
| 0x00 | PING | нет |
| 0x01 | INIT | нет |
| 0x02 | GET_STATUS | нет |
| 0x03 | RESET_RADIO | нет |
| 0x10 | SET_PHY_CONFIG | channel u8, data_rate u8, preamble_length u16, preamble_code u8, PRF u8, PAC_size u8 |
| 0x11 | SET_TX_POWER | power_level u8 |
| 0x20 | TX_FRAME | length u16, payload[length] |
| 0x21 | TX_PERIODIC | period_ms u16, length u16, payload[length] |
| 0x22 | TX_STOP | нет |
| 0x30 | RX_START | нет |
| 0x31 | RX_STOP | нет |
| 0x40 | GET_SIGNAL_METRICS | нет |
| 0x41 | GET_CIR | offset u16, length u16 |
| 0x50 | START_EXPERIMENT | нет |
| 0x51 | STOP_EXPERIMENT | нет |
| 0x60 | TX_SWEEP | channel_start u8, channel_end u8, power_start u8, power_end u8, preamble_length u16 |
| 0x61 | DETECTOR_TEST | number_of_packets u16, tx_power_start u8, tx_power_end u8 |

> Значения `preamble_length`, `PRF`, `PAC_size` передаются как «сырые» числа
> (симв./МГц/симв.); прошивка транслирует их в enum DecaDriver с валидацией.
> Невалидное значение → `INVALID_PARAM`. Таблицы трансляции:
>
> - **channel** (u8): допустимые `{1,2,3,4,5,7}`, пропускается как есть.
> - **data_rate** (u8): **КОД** `{0:110 kbps, 1:850 kbps, 2:6.8 Mbps}` →
>   `DWT_BR_110K/850K/6M8`. Именно код, а не значение в кбит/с: 850 и 6800 не
>   помещаются в u8.
> - **preamble_length** (u16): `64/128/256/512/1024/1536/2048/4096` → `DWT_PLEN_*`.
> - **preamble_code** (u8): `1..24` (широкая валидация; строгая PRF-зависимая —
>   позже). Задаёт и `txCode`, и `rxCode`.
> - **PRF** (u8): число МГц `16/64` → `DWT_PRF_16M/64M`.
> - **PAC_size** (u8): число символов `8/16/32/64` → `DWT_PAC8/16/32/64`.
>
> **nsSFD** (нестандартный SFD) протоколом не передаётся: прошивка ставит его по
> правилу `data_rate == 110 kbps → nsSFD = 1` (иначе 0), по конвенции EVK/DecaWave.
> `sfdTO` на первом этапе = 0 → драйверный дефолт `DWT_SFDTOC_DEF`. `phrMode` =
> `DWT_PHRMODE_STD`.

## 7. Коды STATUS (ответ МКС)

| STATUS | Имя | Описание |
|---|---|---|
| 0x00 | OK | Команда выполнена успешно |
| 0x01 | UNKNOWN_CMD | CMD_ID отсутствует в таблице / обработчик не зарегистрирован |
| 0x02 | INVALID_PARAM | Параметры вне допустимого диапазона |
| 0x03 | RADIO_BUSY | Радиомодуль занят (например, TX уже активен) |
| 0x04 | RADIO_ERROR | DW1000 вернул ошибку |
| 0x05 | BUFFER_OVERFLOW | payload больше допустимого / переполнение буфера |
| 0x06 | TIMEOUT | Таймаут операции (например, ожидание RX) |
| 0x07 | INTERNAL_ERROR | Внутренняя ошибка прошивки (в т.ч. ошибка CRC запроса) |

> При несовпадении CRC входящего пакета прошивка отвечает кодом ошибки
> (`INTERNAL_ERROR`), а не игнорирует пакет молча.

## 8. Формат данных ответа (DATA) — реализованные команды

### PING (0x00) — ✅
DATA: **1 байт `0x00`** (PONG). LEN = 2 (STATUS + 1 байт DATA).

> Приведено к «Бинарный протокол §11»: для PING формат ответа — `OK uint8`
> (один байт данных). API допускал и пустой DATA («или просто код успеха»), но
> берём строгий вариант протокола — 1 байт.

### INIT (0x01) — ✅
DATA: пусто. STATUS = OK, если инициализированы **все** модули DWM1000; иначе
`RADIO_ERROR`. (INIT выполняет reset + загрузку LDE-микрокода + `dwt_initialise`
по каждому модулю.)

### GET_STATUS (0x02) — ✅
DATA (7 байт), из кэша по модулю M1:

| Поле | Тип | Прим. |
|---|---|---|
| TX_state | u8 | пока 0 (трекинга TX/RX ещё нет) |
| RX_state | u8 | пока 0 |
| channel | u8 | из кэша (0 до SET_PHY_CONFIG) |
| data_rate | u8 | из кэша |
| preamble_length | u16 (LE) | из кэша |
| PRF | u8 | из кэша |

### RX_START (0x30) / RX_STOP (0x31) — ✅ 🔬
Параметров нет. DATA нет. `RX_START` включает непрерывный приём на модуле
`DW_RX_LISTEN_DEV` (board_config; по умолчанию M2), требует предварительного INIT
(иначе `RADIO_ERROR`). Приём и разбор кадров — в main loop (`PROTOCOL_PollRadio`,
опрос SYS_STATUS; EXTI добавим позже). `RX_STOP` — `dwt_forcetrxoff`.
Проверено на железе (2026-07-16): M2 расслышал EVK1000 Mode 3, свап на M1 не понадобился.

### GET_SIGNAL_METRICS (0x40) — ✅ 🔬 ⚠️ ИНТЕРИМ-формат
Пока кадра не принято → `TIMEOUT`. Иначе DATA (**18 байт, u16 LE**) — сырые поля
`dwt_rxdiag_t` последнего кадра:

| Смещение | Поле | Источник (dwt_rxdiag_t) |
|---|---|---|
| 0..1 | count (число принятых кадров) | счётчик прошивки (отладка) |
| 2..3 | CIR_PWR | `maxGrowthCIR` |
| 4..5 | RXPACC | `rxPreamCount` |
| 6..7 | STD_NOISE | `stdNoise` |
| 8..9 | FP_AMPL1 | `firstPathAmp1` |
| 10..11 | FP_AMPL2 | `firstPathAmp2` |
| 12..13 | FP_AMPL3 | `firstPathAmp3` |
| 14..15 | FP_INDEX | `firstPath` |
| 16..17 | MAX_NOISE | `maxNoise` |

> ⚠️ **ИНТЕРИМ.** Это сырые поля для проверки факта приёма и как входы для
> формул. Итоговый формат — RSSI int16 (dBm×100), SNR int16 (dB×100), RXPACC u16,
> FP_INDEX u16 — по формулам **DW1000 UM §4.7** (заменит интерим).
>
> **Находка для перехода к RSSI (по замерам EVK Mode 3, преамбула 1024):** RXPACC
> замерен ≈ **261–280** — счётчик насытился рано, т.е. `RXPACC ≠ RXPACC_NOSAT`,
> значит по UM §4.7 SFD-коррекция N **не требуется**. Но строго это подтвердить
> нельзя: `RXPACC_NOSAT` в интерим-формате отсутствует. Для RSSI (UM §4.7.2:
> `RX_LEVEL = 10·log10(C·2^17 / N²) − A`, где `A = 113.77 @PRF16 / 121.74 @PRF64`,
> `C = CIR_PWR`, `N = RXPACC`) **нужно добавить чтение `RXPACC_NOSAT`** (из
> `DRX_CONF`/`RXPACC_NOSAT`), чтобы корректно брать N.

### Форматы ответа запланированных команд (📋, справочно из исходного протокола)

- **GET_CIR:** N = length; пары I/Q int16: I1,Q1,I2,Q2,…,IN,QN.

Эти форматы будут подтверждены/скорректированы при реализации.

## 9. Референс CRC8 (Python)

```python
def crc8(data: bytes) -> int:
    # CRC-8, poly 0x07, init 0x00, no reflection, xorout 0x00
    crc = 0
    for b in data:
        crc ^= b
        for _ in range(8):
            crc = ((crc << 1) ^ 0x07) & 0xFF if (crc & 0x80) else (crc << 1) & 0xFF
    return crc
```

Область: для команды `crc8(bytes([LEN, CMD_ID, *PARAMS]))` — **без SYNC**.

## 10. Примеры кадров

**PING** (запрос):
```
AA 55        SYNC
01           LEN  (CMD_ID + PARAMS = 1)
00           CMD_ID = PING
<crc>        CRC8 по [01, 00]
```
Ответ:
```
AA 55        SYNC
02           LEN  (STATUS + DATA = 2)
00           STATUS = OK
00           DATA   = PONG
<crc>        CRC8 по [02, 00, 00]
```

**GET_STATUS** (запрос): `AA 55 01 02 <crc>` (CRC по `[01,02]`).
