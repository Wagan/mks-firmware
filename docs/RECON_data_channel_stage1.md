# RECON — канал данных, стадия (1): текстовый чат + фрагментация >125 Б

**Тип:** разведка по коду (правки НЕ вносились). **Дата:** 2026-07-22.
**Задача:** оценить фактическое состояние TX-пути, темпа передачи, двусторонности на
одной плате и хостовой сборки/разборки перед реализацией стадии (1) §15.1 — текстовый
чат между двумя МКС с фрагментацией сообщений >125 Б.
**Источник:** `App\protocol\protocol.c` + `.h`, `App\board\board_config.h`,

## 1. TX-путь под фрагментацию — `HandleTX_FRAME` (`protocol.c:982–1014`)

**Приём параметров (wire):** `length u16 LE` (`:989`) + `payload[length]` (`&params[2]`).

**Валидация (отвергает, НЕ обрезает):**
- `params_len < 2` → `INVALID_PARAM` (`:988`) — нет поля length.
- `params_len < 2 + length` → `INVALID_PARAM` (`:990`) — payload короче заявленного.
- **`length > TX_FRAME_MAX(125)` → `BUFFER_OVERFLOW`** (`:991`). Не обрезает, не шлёт.
- модуль-источник не инициализирован / SPI-выбор не удался → `RADIO_ERROR` (`:993–994`).

**Синхронность статуса — ДА, команда блокирующая:**
```c
dwt_writetxdata(length+2, &params[2], 0)   // :999  (иначе RADIO_ERROR)
dwt_writetxfctrl(length+2, 0, 0)           // :1001
dwt_starttx(DWT_START_TX_IMMEDIATE)        // :1003 (иначе RADIO_ERROR)
uint32_t guard = TX_WAIT_GUARD;            // :1007  = 100000 (define :957)
while (!(SYS_STATUS & TXFRS)) {            // :1008  busy-wait завершения TX
    if (--guard == 0) return STATUS_TIMEOUT; // :1009
}
return STATUS_OK;                          // :1013  только ПОСЛЕ ухода кадра
```
Т.е. `TX_FRAME` возвращает статус **только после того, как кадр реально ушёл в эфир**
(`TXFRS`) либо исчерпан guard (`TIMEOUT`). `+2` — резерв под авто-FCS (железо дописывает).

**Несколько кадров подряд:** отдельного «пакетного» механизма в прошивке НЕТ — **каждый
фрагмент = отдельная команда `TX_FRAME`** с хоста, каждая **блокирует** до своего `TXFRS`.
Порядок гарантируется тем, что команды исполняются последовательно в main loop
(`PROTOCOL_PollRx`). ⚠️ `TX_WAIT_GUARD` — **счётчик итераций спина, не время**; на
медленном Mode 3 (110 кбит/с) полный 125-Б кадр — это заметное эфирное время, но защита
от вечного цикла есть. На время блокировки `TX_FRAME` main loop занят (RX/поток не
обслуживаются до возврата) — для коротких серий фрагментов приемлемо; пейсинг между
фрагментами (если понадобится) — на хосте.

## 2. TX_PERIODIC как образец темпа — `HandleTX_PERIODIC` + `PROTOCOL_PollTx`

**Взвод режима — `HandleTX_PERIODIC` (`protocol.c:1038–1059`):**
- Params: `period_ms u16 LE` (`:1045`), `length u16 LE` (`:1046`), `payload[length]`.
- Валидация: `params_len<4` → `INVALID_PARAM`; `length>125` → `BUFFER_OVERFLOW` (`:1047`);
  `params_len<4+length` → `INVALID_PARAM` (`:1048`); `period<TX_PERIOD_MIN_MS(5)` →
  `INVALID_PARAM` (`:1049`); не инициализирован → `RADIO_ERROR` (`:1050`).
- **Хранение payload:** `memcpy(tx_periodic_frame, &params[4], length)` (`:1052`), буфер
  `static uint8_t tx_periodic_frame[TX_FRAME_MAX]` (`:973`). `tx_periodic_len=length`,
  `tx_period_ms=period`, `tx_last_ms = now - period` (первая посылка сразу),
  `tx_periodic_active=1` (`:1053–1057`).
- **Команда возвращает `OK` сразу** (только взводит режим); реальная посылка — в PollTx.

**Реальная посылка — `PROTOCOL_PollTx` (`protocol.c:1274–1297`, main loop):**
- `if (!tx_periodic_active) return` (`:1276`); интервал по `HAL_GetTick()`:
  `if (now - tx_last_ms < tx_period_ms) return` (`:1279`).
- По достижении периода: выбрать `DW_TX_SOURCE_DEV`, `writetxdata(tx_periodic_frame)`,
  `writetxfctrl`, `starttx`, busy-wait `TXFRS` с guard (`:1282–1296`) — тем же паттерном,
  что `TX_FRAME`. `tx_periodic_count++`.

**Вывод:** механизм «плата САМА шлёт последовательность кадров» уже есть, НО он
**повторяет ОДИН И ТОТ ЖЕ фиксированный кадр** (`tx_periodic_frame`) с интервалом — это
не последовательность РАЗНЫХ фрагментов. Для сообщения из разных фрагментов текущий
`TX_PERIODIC` не подходит: под каждый фрагмент нужен свой payload. Текущий рабочий путь
под фрагменты — **хост шлёт N команд `TX_FRAME`** (по фрагменту на команду, §1).
Расширять ли прошивку под «пакет фрагментов одной командой» — архитектурное решение.

## 3. Двусторонность на одной плате — роли и одновременный TX+RX

**Роли фиксированы в `board_config.h` (BOARD_MKS, `:31–46`):**
- `DW_DEVICE_COUNT = 2` (`:31`); `DW_DEV_M1 = 0`, `DW_DEV_M2 = 1` (`:37–38`).
- `DW_RX_LISTEN_DEV = DW_DEV_M2` (`:42`) — приём.
- `DW_TX_SOURCE_DEV = DW_DEV_M1` (`:46`) — передача.
- Роли — **compile-time `#define`**, на одной плате M1 всегда TX, M2 всегда RX.

**И принимать (M2), и передавать (M1) в одной сессии без переинициализации — ДА
(подтверждено):**
- Переключение активного чипа — на КАЖДУЮ операцию: `deca_port_select_device(idx)`
  (`dwt_setlocaldataptr` + шина/CS). RX-путь: `PROTOCOL_PollRadio` выбирает
  `DW_RX_LISTEN_DEV` (`:1211`) и **перевзводит приём** после каждого кадра
  (`dwt_rxenable`, `:1237`). TX-путь: `HandleTX_FRAME`/`PollTx` выбирают
  `DW_TX_SOURCE_DEV` (`:994`, `:1282`).
- Всё — в **main loop, последовательно** (`PollRx`/`PollRadio`/`PollTx` без вытеснения),
  поэтому TX и RX не конфликтуют за SPI/состояние.
- **TX во время активного RX-потока НЕ роняет приём — подтверждено приёмкой стадии (0):**
  loopback M1(TX по `txperiodic`)→M2(RX+поток content=3) отдавал данные байт-в-байт,
  SEQ рос, dropped=0. После каждого TX следующий `PollRadio` заново выбирает RX-модуль
  и перевзводит приём.

**Для чата «двух МКС»:** механизм ролей уже поддерживает одновременный TX(M1)+RX(M2) —
**правок `board_config` не требуется**. Отличие от loopback только в том, что кадры
летят между ДВУМЯ платами (M1 одной → M2 другой), а не внутри одной. M1/M2 — раздельные
SPI-шины и антенны.

## 4. Хостовая сторона — сборка/разборка сообщений

**Разбор приёма (`tools\mks_stream.py`, `parse_stream_body`):**
- `content=3` → возвращает `{seq, dropped, content, metrics:None, cir:None, data:bytes}`
  — **данные ОДНОГО кадра** (тело без FCS). Устойчив к усечению (`ProtocolError`).

**Приёмник (`tools\mks_data_probe.py`):** читает кадры, печатает/сверяет **по-кадрово**;
поле каждого кадра — `fr["data"]`. **Склейки нескольких `data`-кадров в одно сообщение
НЕТ** — реассемблинг сообщения (по будущему заголовку фрагментации) писать **с нуля**.

**Отправка (хост→радио):**
- `tools\mks_protocol.py`: `tx_frame(payload)` (`:200–206`) — пакует `length u16 LE` +
  payload, шлёт `TX_FRAME`. **Пред-проверки 125 на хосте НЕТ** (только `>65535` →
  `ValueError`); при `>125` прошивка вернёт `BUFFER_OVERFLOW`. `tx_periodic(period,
  payload)` (`:215–227`).
- `tools\mks_console.py`: `cmd_txframe` (`:260`) / `cmd_txperiodic` (`:277`) — парсят hex
  и зовут методы. Удобная отправка ОДНОГО кадра есть; **нарезка сообщения >125 на кадры
  и сборка на приёме — новый код стадии (1)**.

**Итог:** транспорт (послать один кадр / принять один `data`-кадр) готов; прикладной
слой (фрагментация при отправке + реассемблинг при приёме + UTF-8) — **писать с нуля,
на хосте** (прошивка остаётся прозрачным мостом, фундамент стадии 0 достаточен).

## 5. Граница фрагмента (арифметика; заголовок НЕ придумываю)

- **Максимум полезных байт в одном кадре на проводе = 125** (`TX_FRAME_MAX`, `:956`);
  приём отдаёт ровно `frame_len − 2` ≤ 125 (стандартный кадр). Round-trip не теряет.
- Заголовок фрагментации (seq/total/flags — **решение архитектора**) живёт **ВНУТРИ**
  этих 125 байт (прикладной слой поверх кадра). Значит под данные фрагмента остаётся:
  **`125 − N`**, где `N` — размер заголовка фрагмента.
  - Ориентир (НЕ выбор формата): `N=3` → **122 Б**; `N=4` → **121 Б**; `N=6` → **119 Б**.
- Число фрагментов сообщения `M` байт: `ceil(M / (125 − N))`. Пример при `N=4`,
  сообщение 300 Б → `ceil(300/121) = 3` фрагмента.
- Заголовок и его поля — **за архитектором** (по итогам этого отчёта).

## 6. Расхождения с PROTOCOL_SPEC v1.5

- **Материальных расхождений НЕТ.** `TX_FRAME` (лимит 125 → `BUFFER_OVERFLOW`),
  `TX_PERIODIC` (взвод + PollTx), `SET_STREAM_MODE content=3` — совпадают с §8.
- **Хостовая заметка (не SPEC):** `mks_protocol.tx_frame` не проверяет 125 до отправки —
  при фрагментации хост ОБЯЗАН резать на ≤125 сам, иначе `BUFFER_OVERFLOW` от прошивки.

## 7. Вывод для стадии (1)
- **Прошивку менять НЕ обязательно:** транспорт (TX одного кадра ≤125 синхронно; RX →
  `content=3` данные байт-в-байт; одновременный TX+RX на одной плате) — готов.
- **Всё новое — на хосте:** (а) нарезка сообщения на фрагменты ≤`125−N`; (б) заголовок
  фрагмента (архитектор); (в) отправка фрагментов серией `tx_frame`; (г) реассемблинг
  на приёме из `content=3` `data`-кадров; (д) UTF-8 упаковка/распаковка; (е) простой
  чат-UI (2 МКС / 2 COM-порта, либо loopback для отладки).
- **Открытые решения архитектора:** формат заголовка фрагментации (N байт, поля);
  надёжность на этой стадии (пока БЕЗ ARQ — это стадия 2) → потерянный фрагмент =
  битое сообщение (для демонстрации фрагментации приемлемо); нужен ли пейсинг между
  фрагментами на медленном Mode.
