# Отчёт: диагностика зависания INIT (HAL_Delay в контексте USB-прерывания)

**Дата:** 2026-07-16
**Симптом:** PING отвечает мгновенно, INIT не отвечает (>30 с = вечно).
**Вердикт:** гипотеза **ПОДТВЕРЖДЕНА**. INIT виснет в `HAL_Delay()`, вызванном
из контекста USB-прерывания, где `SysTick` не может прервать USB IRQ, поэтому
`uwTick` не инкрементируется и `HAL_Delay` крутится бесконечно.
**Правки:** не вносились (по договорённости — сначала согласовать план).

---

## 1. Из какого контекста вызываются обработчики

Обработка команды идёт **в контексте USB-прерывания** (ISR), а не в main loop.
Цепочка вызовов:

```
OTG_FS_IRQHandler()                       Core/Src/stm32f4xx_it.c:204   (ISR)
  → HAL_PCD_IRQHandler(&hpcd_USB_OTG_FS)  stm32f4xx_it.c:209
    → (USB стек) USBD_CDC_DataOut()        Middlewares/.../usbd_cdc.c:577
      → ->Receive() == CDC_Receive_FS()    usbd_cdc.c:592  (наш callback)
        → PROTOCOL_ProcessByte()           USB_DEVICE/App/usbd_cdc_if.c (USER CODE 6)
          → handlers[CMD_INIT] → HandleINIT()   App/protocol/protocol.c
            → deca_port_select_device(i)
            → deca_port_hard_reset(i)       App/platform/deca_port.c:87-90
                → deca_sleep(2) → HAL_Delay(2)   ← ВИСНЕТ УЖЕ ЗДЕСЬ
            → dwt_initialise(DWT_LOADUCODE) → (deca_sleep/HAL_Delay внутри)
```

Т.е. `CDC_Receive_FS` исполняется **из USB IRQ**, и весь разбор+диспетчеризация+
обработчик выполняются синхронно в этом прерывании.

## 2. Приоритеты NVIC — SysTick не может прервать USB IRQ

| Прерывание | Preempt priority | Где задано |
|---|---|---|
| Группировка | `NVIC_PRIORITYGROUP_4` (4 бита preempt, 0 sub) | `stm32f4xx_hal.c:173` |
| **OTG_FS_IRQn** | **0** (высший) | `USB_DEVICE/Target/usbd_conf.c:94` — `HAL_NVIC_SetPriority(OTG_FS_IRQn, 0, 0)` |
| **SysTick** | **15** (низший) | `TICK_INT_PRIORITY = 15U`, `stm32f4xx_hal_conf.h:151` |

На Cortex-M меньшее число = выше приоритет. USB IRQ = 0, SysTick = 15 →
**SysTick НЕ может вытеснить USB IRQ.** Пока МК находится внутри USB-прерывания,
`SysTick_Handler` (`stm32f4xx_it.c:183`) не выполняется, значит `uwTick` заморожен.
`HAL_Delay(ms)` крутит `while ((uwTick - tickstart) < wait)` — условие никогда не
станет истинным → **бесконечный цикл в ISR**. Ядро застревает в USB-прерывании,
main loop не возобновляется. Это и есть «зависание».

## 3. Подтверждения по пунктам

- **`deca_sleep` = `HAL_Delay`:** `App/platform/deca_port.c:171-173`
  (`void deca_sleep(unsigned int time_ms) { HAL_Delay(time_ms); }`). ✔
- **`HandleINIT` вызывает всё синхронно** в этом же контексте:
  `deca_port_select_device` → `deca_port_hard_reset` → `dwt_initialise`. ✔
- **Первый блокирующий вызов — ещё в `deca_port_hard_reset`:** `deca_sleep(2)` на
  строках `deca_port.c:88` и `:90` (до сброса и после). Т.е. виснет **до**
  `dwt_initialise`, на аппаратном сбросе. `dwt_initialise` (LDE-загрузка,
  softreset) содержит ещё `deca_sleep` — тоже повис бы. ✔
- **Почему PING работает:** `HandlePING` — чистое вычисление + `CDC_Transmit_FS`,
  без `HAL_Delay`; возвращается мгновенно, ISR завершается штатно. ✔

## 4. Косвенное подтверждение

Стартовый `bringup_read_devids()` (тот же `hard_reset`+`deca_sleep`+`dwt_readdevid`)
**работает** — но он вызывается из `main()` в **thread mode** (до `while(1)`,
`Core/Src/main.c` USER CODE 2), где `SysTick` работает и `HAL_Delay` тикает.
Разница ровно в контексте исполнения — что подтверждает диагноз.

## 5. Предлагаемая архитектура (тяжёлые команды — вне ISR)

**Принцип:** ISR приёма USB не должен исполнять блокирующие/долгие операции.
ISR только принимает байты; разбор и обработчики — в main loop (thread mode),
где `HAL_Delay`/`deca_sleep` работают. Это чинит INIT и все будущие блокирующие
команды (TX/RX, softreset, LDE-загрузка после сна и т.д.).

### Вариант B (рекомендуемый) — сырой RX-кольцевой буфер, разбор в main loop
- `CDC_Receive_FS` (ISR): только копирует принятые байты в кольцевой буфер
  `rx_ring[]` и пере-вооружает эндпоинт (`USBD_CDC_ReceivePacket`). Ничего не
  парсит, не вызывает обработчики. Быстро, без блокировок.
- `main loop`: `while(1)` вызывает `PROTOCOL_PollRx()`, который выгребает байты
  из `rx_ring[]` и прогоняет их через `PROTOCOL_ProcessByte()`. Теперь и парсер,
  и обработчики, и `HAL_Delay` исполняются в thread mode.
- Кольцевой буфер разделяется ISR↔thread: индексы `volatile`, доступ атомарный
  (u8/u16), для F4 достаточно; при желании — короткая критическая секция.
- Плюсы: минимум логики в ISR; парсер не трогаем; естественно решает и будущие
  блокирующие команды. Классический паттерн «ISR пишет — поток разбирает».

### Вариант A (альтернатива) — декод в ISR, исполнение в main loop
- Парсер работает в ISR как сейчас, но по готовности валидного пакета НЕ
  вызывает обработчик, а кладёт декодированную команду (cmd_id + params) в
  один слот/маленькую очередь и ставит `volatile` флаг.
- `main loop`: видит флаг → исполняет обработчик + шлёт ответ (thread mode).
- Минус: CRC-разбор всё ещё в ISR (быстрый, но лишняя логика в прерывании);
  очередь команд усложняет структуру. Для одиночных команд MATLAB избыточно.

**Рекомендация:** Вариант **B** — проще, надёжнее, дешевле по логике в ISR.

### Сопутствующее (заметки, не блокеры)
- **`CDC_Transmit_FS` может вернуть `USBD_BUSY`.** В main loop это легко
  пережить (подождать/повторить); сейчас — одна попытка. Отдельным пунктом.
- **Переполнение `rx_ring`:** при быстром потоке команд. Для MATLAB (редкие
  мелкие пакеты) буфера 256–512 Б хватает; предусмотреть флаг overflow.
- **IWDG (позже):** когда введём watchdog — долгий INIT в main loop должен
  укладываться в окно либо гладить собаку. Пока IWDG нет — не блокер.
