# RECON — RSSI-в-beacon: количественный индикатор связи (§15.1)

**Тип:** разведка по коду (правки НЕ вносились). **Дата:** 2026-07-22.
**Задача:** понять, как станция (`content=3`, только данные) может получить RSSI
принятого от партнёра кадра, не ломая чат. **Источник:** `App\protocol\protocol.c`,
`tools\mks_stream.py`, `tools\mks_station.py`, `tools\mks_protocol.py`.
**Сверено с** `PROTOCOL_SPEC.md` v1.5 §8. Только факты и стоимости — выбор за архитектором.

---

## 1. Метрики и данные — один и тот же принятый кадр (ПОДТВЕРЖДЕНО)

**`PROTOCOL_PollRadio`, ветка RXFCG (`protocol.c:1215–1237`):** на ОДИН хороший кадр
последовательно:
```c
frame_len = RX_FINFO & RXFL;                 // :1219
dwt_readrxdata(rx_frame, frame_len, 0);      // :1222  ТЕЛО кадра -> rx_frame
dwt_readdiagnostics(&rx_metrics.diag);       // :1225  МЕТРИКИ этого же кадра
rx_metrics.rxpacc_nosat = ...;               // :1227
rx_metrics.valid = 1; rx_metrics.count++;    // :1228-1229
PROTOCOL_CaptureCIR();                        // :1231
if (stream_active) PROTOCOL_SendStreamFrame();// :1234  ОДИН потоковый кадр
dwt_rxenable(...);                            // :1237
```
- `build_metrics_block` (`:814–847`) читает **`rx_metrics.diag`** — диагностику кадра,
  только что прочитанного в `rx_frame`. `SendStreamFrame` для `content=3` шлёт тело
  `rx_frame` (стадия 0). ⇒ **метрики и данные в потоковом кадре относятся к ОДНОМУ
  принятому RXFCG-кадру.** Один RXFCG → один `SendStreamFrame` (`:1234`).
- **Критично (без домыслов): `rx_metrics` перезаписывается на КАЖДОМ RXFCG** —
  и на партнёрском кадре, И на СВОЁМ эхо (М2 слышит свой М1). В прошивке **нет
  привязки метрик к источнику** (src_id — это прикладной слой хоста, прошивке
  неизвестен). Т.е. «последние метрики» могут быть от собственного эха, не от партнёра.

## 2. Стоимость `content=4` «данные + метрики» (правка ПРОШИВКИ)

Гипотетический `content=4` = тело кадра (как `content=3`) + блок метрик 30 Б (как
`content=2`), CIR не нужен. Объём правки — **малый, 3 точки** (по образцу `content=3`):

1. **Порог валидации** — `HandleSET_STREAM_MODE` (`protocol.c:941`): сейчас
   `if (mode > 3) return INVALID_PARAM;` → поднять до `mode > 4`. Присваивание
   `stream_content = mode` (`:947`) уже обобщённое.
2. **Ветка сборки** — `PROTOCOL_SendStreamFrame` (`:1120–...`): по образцу текущих
   веток — вписать `data_len(u16 LE)+data` (как `content=3`) **и** `build_metrics_block`
   (30 Б, как `content=1/2`); порядок полей — решение архитектора (напр. метрики,
   затем data_len+data). Ранний guard на `frame_len<2 / >128` — как у `content=3`.
3. **Хост** — `mks_stream.parse_stream_body` (`tools\mks_stream.py`): добавить ветку
   `content==4` (разобрать И метрики 30 Б, И data_len+data). Сейчас ветки: 1/2 →
   метрики(+CIR), 3 → data (`:37–...`).

**Бюджет буфера — влезает с запасом.** `frame[320]` (`:1122`). Максимум `content=4`:
```
SMARK 2 + LEN16 2 + SEQ 2 + DROPPED 2 + CONTENT 1 + метрики 30
        + data_len 2 + data (<=126) + CRC 1  =  168 байт   (<= 320, запас ~152)
```
CIR (до 252 Б) не входит — поэтому `content=4` заметно короче `content=1` (298 Б).

**Ключевое преимущество `content=4` для замера партнёра:** метрики едут ВМЕСТЕ с телом
каждого кадра, поэтому станция применяет свой УЖЕ существующий эхо-фильтр (отбросить
`src==own`, см. §5) и берёт RSSI **только из партнёрских кадров** → корректная
привязка «RSSI ↔ партнёр». (В отличие от команды 0x40, §3.)

## 3. Альтернатива без прошивки: команда `GET_SIGNAL_METRICS` (0x40) во время потока

**Различимость форматов на хосте — ДА.** Командный ответ и потоковый кадр имеют РАЗНЫЕ
маркеры и разбираются разными ридерами:
- командный ответ: `SYNC 0xAA55` — `MKS.read_response` ресинкается по `0xAA 0x55`
  (`mks_protocol.py:163–172`);
- потоковый кадр: `SMARK 0xDE 0xCA` — `StreamReader` ресинкается по SMARK
  (`mks_stream.py`, `find(SMARK)`).
Форматы **отличимы по маркеру** — хост не спутает.

**Firmware НЕ блокирует команды во время потока (по коду).** `PROTOCOL_PollRx`
(разбор команд) и `PROTOCOL_PollRadio` (эмиссия потока на RXFCG) — оба в main loop;
взаимного лока нет. Команда `0x40` будет обработана и отвечена `SYNC`-кадром, пока
`SMARK`-кадры продолжают идти. (Заметка `mks_protocol.py:262–263` «командный канал
занят потоком» — это **хостовая предосторожность**, не firmware-лок.)

**НО практические проблемы (факты):**
- `MKS.command` (`:177–180`) делает `flush_input()` → **сбрасывает буфер OS**, теряя
  накопленные потоковые кадры; при чтении ответа `read_response` посимвольно ищет
  `0xAA55` среди приходящих SMARK-байт — есть риск **ложного `0xAA55` внутри тела
  потокового кадра** → CRC ответа не сойдётся → `ProtocolError`.
- Один владелец порта: станция в `engine`-треде должна **вклинивать** команду между
  `reader.poll()` (как TX в GUI-паттерне), + `reader.buf.clear()` после.
- **Главное — привязка к источнику отсутствует:** `GET_SIGNAL_METRICS` возвращает
  `rx_metrics` = **последний RXFCG-кадр**, который может быть СВОИМ эхом (§1), а не
  партнёром. Хост **не может определить**, чьи это метрики. Для замера «как слышно
  партнёра» это ненадёжно (в отличие от `content=4`, где метрики привязаны к кадру и
  фильтруются эхо-фильтром по src).

## 4. Какие поля есть в блоке метрик (30 Б; `build_metrics_block:814–847` / SPEC §8)

Все — LE; знаковые в i16-контейнере (`INT16_MIN` = «н/д»):
| Поле | Тип | Смысл |
|---|---|---|
| count | u16 | счётчик принятых кадров (отладка) |
| CIR_PWR | u16 | `maxGrowthCIR` |
| RXPACC | u16 | `rxPreamCount` (сырой) |
| STD_NOISE | u16 | `stdNoise` |
| FP_AMPL1..3 | u16×3 | амплитуды первого пути |
| FP_INDEX | u16 | `firstPath` (сырой 10.6 fixed-point) |
| MAX_NOISE | u16 | `maxNoise` |
| RXPACC_NOSAT | u16 | для SFD-коррекции N |
| N_corrected | u16 | N после коррекции |
| **RSSI (RX_LEVEL)** | i16 | **dBm×100 — общий уровень принятого сигнала** (UM §4.7.2) |
| **FP_POWER** | i16 | **dBm×100 — мощность первого пути** |
| A_used | u16 | константа A ×100 |
| **SNR** | i16 | dB×100 (= RSL + delta) |

Для замера «насколько партнёр далеко/слабо» физически осмысленны два поля мощности:
**RSSI (RX_LEVEL)** — общий уровень (интегральный, включая многолучёвку) и **FP_POWER**
— мощность прямого (первого) пути. Разница `RSSI − FP_POWER` — это LOS/NLOS-признак
(§8 SPEC). **Выбор поля — за архитектором** (просто перечислил что есть).

## 5. Куда RSSI попадёт в станции (чисто хостовая привязка)

**Текущее состояние партнёра — `Station.__init__` (`mks_station.py:103–105`):**
```python
self.last_heard = 0.0        # монотонное время последнего ЧУЖОГО кадра
self.last_heard_src = None
self.link_up = False
```
**Обновляется в `engine` (`:186–187`)** — ПОСЛЕ эхо-фильтра (`:183` отбрасывает
`is_own_echo`), т.е. только на партнёрских кадрах:
```python
self.last_heard = now
self.last_heard_src = f["src_id"]
```
**Отображается в `status_line` (`:228–237`)** и `_update_link` (`:215–226`); сейчас
строка статуса содержит буквально `RSSI: н/д (content=3)`.

**Логичное место для RSSI:** рядом с `last_heard_src` завести поле (напр.
`self.last_rssi`, при желании `last_fp`/`last_snr`), обновлять в том же месте `:186–187`
(на партнёрском кадре) и печатать в `status_line`/индикаторе. Это возможно только если
метрики ДОХОДЯТ до станции с привязкой к кадру:
- при `content=4` — метрики уже в том же кадре, `parse_stream_body` их вернёт, привязка
  к src через эхо-фильтр (§2);
- при команде `0x40` — привязки к src нет (§3), поле обновлялось бы «последними любыми»
  метриками (в т.ч. своё эхо).

## 6. Расхождения с SPEC v1.5

- **`content=4` в SPEC НЕТ** (не реализован — расхождения сейчас нет). При реализации
  §8 SET_STREAM_MODE дополнить (mode=4, формат PAYLOAD метрики+data).
- **Заметка про «командный канал занят потоком»** (`mks_protocol.py:262–263`) — это
  хостовая формулировка; **в прошивке лока команд во время потока НЕТ** (§3). Строгого
  противоречия со SPEC нет, но формулировка сильнее, чем реальное поведение firmware.
- Прочее (метрики 30 Б, RXFCG→один потоковый кадр, маркеры SYNC/SMARK) — совпадает.
