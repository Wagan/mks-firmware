## 1. Цель (минимальный рабочий срез под НИР)

Графическое приложение: подключиться к плате МКС по COM-порту, выбрать PHY-режим,
включить приём, и показывать **факт присутствия СШП в канале** (принят кадр →
сигнал есть) + живой график CIR.

Детектор в этой версии — **простой**: «за последнюю секунду принят хотя бы один
кадр → СШП ЕСТЬ». Пороговый детектор с P_false — отдельная будущая задача, НЕ здесь.

Всё поверх уже реализованных и проверенных на железе команд: INIT (0x01),
SET_PHY_CONFIG (0x10), RX_START (0x30), RX_STOP (0x31), GET_SIGNAL_METRICS (0x40),
GET_CIR (0x41). Ничего нового в протоколе не нужно.

## 2. Почему выбор режима, а не фиксированный Mode 3 (контекст)

DW1000 принимает кадр только если приёмник настроен на ТОТ ЖЕ комплексный канал,
что и передатчик (channel, PRF, preamble code, data rate, SFD). На чужом режиме
RXFCG не сработает → детектор ложно скажет «нет сигнала». Поэтому GUI должен давать
выбор PHY-режима (пресеты Mode 1–8 + ручной), а не хардкодить Mode 3.

Полноценный автосканер по режимам (перебор каналов/PRF/преамбул) — БУДУЩАЯ задача,
в этот GUI НЕ входит. Здесь: пользователь выбирает режим вручную, плата слушает на нём.

## 3. Таблица PHY-режимов EVK1000 (пресеты — из первоисточника, НЕ по памяти)

Снято из исходников EVK1000 (`dw_main.c`, массив `chConfig[8]`). Наш `SET_PHY_CONFIG`
принимает по проводу: `channel u8, data_rate u8 (КОД), preamble_length u16,
preamble_code u8, PRF u8 (МГц), PAC_size u8`. Порядок для консольного `setphy`:
`<ch> <dr> <plen> <code> <prf> <pac>`.

data_rate КОД: `0 = 110 kbps`, `1 = 850 kbps`, `2 = 6.8 Mbps`.
PRF передаётся как число МГц: `16` или `64`.
nsSFD по проводу НЕ передаётся — прошивка сама ставит `nsSFD = (data_rate==110k)?1:0`.
Сверено с таблицей: 110k-режимы → nsSFD=1, 6M8-режимы → nsSFD=0 — совпадает.

| Mode | channel | data_rate (код) | preamble_len | preamble_code | PRF (МГц) | PAC | setphy-строка |
|------|---------|-----------------|--------------|---------------|-----------|-----|---------------|
| 1    | 2       | 110k (0)        | 1024         | 3             | 16        | 32  | `2 0 1024 3 16 32` |
| 2    | 2       | 6M8 (2)         | 128          | 3             | 16        | 8   | `2 2 128 3 16 8`   |
| 3    | 2       | 110k (0)        | 1024         | 9             | 64        | 32  | `2 0 1024 9 64 32` |
| 4    | 2       | 6M8 (2)         | 128          | 9             | 64        | 8   | `2 2 128 9 64 8`   |
| 5    | 5       | 110k (0)        | 1024         | 3             | 16        | 32  | `5 0 1024 3 16 32` |
| 6    | 5       | 6M8 (2)         | 128          | 3             | 16        | 8   | `5 2 128 3 16 8`   |
| 7    | 5       | 110k (0)        | 1024         | 9             | 64        | 32  | `5 0 1024 9 64 32` |
| 8    | 5       | 6M8 (2)         | 128          | 9             | 64        | 8   | `5 2 128 9 64 8`   |

Mode 3 = наш эталон (совпадает с консольным `mode3`). Дефолт в GUI при старте — Mode 3.

В коде GUI объявить эти пресеты как таблицу (список словарей), пример:
```python
PHY_MODES = {
    "Mode 1 (ch2, 110k, PRF16, code3)":  dict(ch=2, dr=0, plen=1024, code=3, prf=16, pac=32),
    "Mode 2 (ch2, 6M8, PRF16, code3)":   dict(ch=2, dr=2, plen=128,  code=3, prf=16, pac=8),
    "Mode 3 (ch2, 110k, PRF64, code9)":  dict(ch=2, dr=0, plen=1024, code=9, prf=64, pac=32),
    "Mode 4 (ch2, 6M8, PRF64, code9)":   dict(ch=2, dr=2, plen=128,  code=9, prf=64, pac=8),
    "Mode 5 (ch5, 110k, PRF16, code3)":  dict(ch=5, dr=0, plen=1024, code=3, prf=16, pac=32),
    "Mode 6 (ch5, 6M8, PRF16, code3)":   dict(ch=5, dr=2, plen=128,  code=3, prf=16, pac=8),
    "Mode 7 (ch5, 110k, PRF64, code9)":  dict(ch=5, dr=0, plen=1024, code=9, prf=64, pac=32),
    "Mode 8 (ch5, 6M8, PRF64, code9)":   dict(ch=5, dr=2, plen=128,  code=9, prf=64, pac=8),
}
```

Плюс режим «Ручной» — шесть полей ввода (ch, dr, plen, code, prf, pac), отдаются в
SET_PHY_CONFIG как есть. Валидацию диапазонов делает прошивка (вернёт INVALID_PARAM).

## 4. Функциональность GUI (минимум под НИР)

### 4.1. Подключение
- Поле/выпадающий список COM-порта (можно просто текстовое поле «COM3» — или
  перечислить порты через `serial.tools.list_ports`, если несложно; не обязательно).
- Кнопка «Подключить» / «Отключить». При подключении — создать `mks.MKS(port)`.
- Индикация состояния подключения (подключено/нет).

### 4.2. Управление
- Кнопка «INIT» — вызвать `dev.init()` (долгая, таймаут больше — как в консоли ~20 с).
- Панель PHY: выпадающий список пресетов (§3) + вариант «Ручной» с 6 полями.
  Кнопка «Применить режим» — собрать params и вызвать SET_PHY_CONFIG (0x10).
  (Параметры собираются ТОЧНО как в консольном `cmd_setphy`:
  `bytes([ch, dr, plen&0xFF, plen>>8, code, prf, pac])`.)
- Кнопки «Старт детекции» / «Стоп»:
  - Старт: `dev.rx_start()`, запустить фоновый поток опроса (см. §5).
  - Стоп: остановить поток, `dev.rx_stop()`.

### 4.3. Индикатор присутствия (главное для НИР)
- Крупная панель/«лампа»: **СШП В КАНАЛЕ — ЕСТЬ** (зелёный фон) / **НЕТ** (серый).
- Логика: хранить время последнего РОСТА счётчика `count` (из GET_SIGNAL_METRICS).
  Если с момента последнего нового кадра прошло < 1.0 с → ЕСТЬ, иначе НЕТ.
  (`count` монотонно растёт при приёме; его прирост = «пришёл новый кадр».)

### 4.4. Счётчики/телеметрия (текстом, обновляются в реальном времени)
- Принято кадров (count).
- Кадров/с (оценка: приращение count за интервал опроса).
- Последний SNR (dB), RSSI (dBm), FP_POWER (dBm) — из финального формата метрик
  (`parse_signal_metrics`, поля snr_db/rssi_dbm/fp_power_dbm; учесть флаги
  *_valid — если невалидно, показывать «н/д»).
- FP_INDEX (из метрик, поле FP_INDEX — сырой; или из CIR fp_index — он уже в
  отсчётах; для отображения взять fp_index из GET_CIR, он нагляднее).
- Текущий режим (какой пресет применён).

### 4.5. Окно CIR (matplotlib)
- Встроенный график (FigureCanvasTkAgg): по X — индекс отсчёта (start_index..),
  по Y — амплитуда `sqrt(I^2+Q^2)` из `parse_cir`. Вертикальная линия/маркер на
  `fp_index` (first path).
- Обновляется по данным `GET_CIR` (half по умолчанию = 0 → дефолт прошивки; можно
  дать поле «полуширина», 0..30, опционально).
- Кнопка «Сохранить PNG» — `figure.savefig(...)` (для НИР-отчёта). Имя файла с
  датой/временем.
- CIR приходит из того же приёма, что метрики; обновлять график с той же частотой,
  что и телеметрию (или чуть реже, чтобы не грузить GUI — напр. раз в 0.3–0.5 с).

## 5. КРИТИЧНО: опрос платы в фоновом потоке (чтобы окно не висло)

tkinter однопоточный. Если опрашивать `dev.get_signal_metrics()`/`get_cir()` прямо
в обработчике кнопки или в главном цикле — окно замёрзнет на время I/O.

**Паттерн (заложить обязательно):**
- Фоновый `threading.Thread` (daemon): в цикле, пока `running`, зовёт
  `dev.get_signal_metrics()` и `dev.get_cir()`, парсит, и складывает результат в
  потокобезопасную очередь `queue.Queue` (или в переменные под `threading.Lock`).
  Между итерациями — небольшой sleep (напр. 0.1–0.2 с), чтобы не забить порт.
- Главный поток (GUI): периодически (`root.after(100, ...)`) забирает последний
  результат из очереди и обновляет виджеты (лампу, счётчики, график). Всё рисование
  — ТОЛЬКО в главном потоке (tkinter не потокобезопасен для отрисовки).
- Обращения к `dev` (порт) — ТОЛЬКО из фонового потока (или сериализовать), чтобы
  не было гонок на COM-порту. Кнопки INIT/SET_PHY/rx_start/rx_stop, если жмутся во
  время работы потока, — синхронизировать (напр. на время INIT приостановить опрос,
  либо все обращения к `dev` гнать через один поток/очередь команд). Простейшее
  безопасное решение: пока идёт детекция (поток активен) — кнопки конфигурации
  (INIT/Применить режим) заблокировать; менять режим только на паузе. Это допустимо
  для первой версии.
- Ошибки порта в потоке (ProtocolError, SerialException) — не ронять GUI: поймать,
  показать в статусной строке, при обрыве — аккуратно остановить детекцию.

Аккуратное закрытие окна: по `WM_DELETE_WINDOW` — остановить поток, `dev.rx_stop()`,
`dev.close()`, затем `root.destroy()`.

## 7. Запуск (документировать в docstring GUI)

```
python mks_gui.py            # порт выбирается в окне
python mks_gui.py COM3       # опционально: порт аргументом, автозаполнить поле
```

Типовой сценарий пользователя (для НИР, отразить в подсказке/README-комментарии):
1. Подключить (COM3).
2. INIT.
3. Выбрать режим (по умолчанию Mode 3) → Применить режим.
4. Старт детекции.
5. Смотреть лампу «СШП В КАНАЛЕ» + график CIR. При наличии совместимого источника
   (киты EVK в том же режиме) — лампа зелёная, CIR живой, счётчики растут.
6. Сменить режим (если источник в другом) → на паузе: Стоп → Применить режим → Старт.
